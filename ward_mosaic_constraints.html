1. Why Ward: escaping the Leiden dead end

The Leiden approach was abandoned after confirming a fundamental structural mismatch: Leiden uses a graph where community membership propagates through multi-hop paths. Suppressing a direct spatial edge (i, j) does not break the indirect path i → k → j through a shared featurally similar neighbor. Because leidenalg also rejects negative edge weights, true repulsion is impossible — clamping to zero just removes a direct edge, leaving indirect paths untouched.

Every modification to Leiden's graph (hard removal V1, multiplicative V2, subtractive-clamped V3, coverage fraction, proximity score) showed ARI ≈ 0.891 on anisotropic — identical to unmodified GMM. The spatial signal is simply absorbed by graph connectivity.
Why Ward doesn't have the path transitivity problem Ward is an agglomerative algorithm: at each step it evaluates the cost of merging exactly two clusters, using a closed-form formula that depends only on the two cluster means and sizes. There is no graph, no multi-hop propagation, no indirect paths. The spatial penalty is evaluated on the pair being merged right now — it cannot be bypassed by routing through a third cluster.

Additional reasons Ward is a natural fit:

  • Objective is variance: Ward minimizes within-cluster variance, which is directly interpretable as "keep featurally similar cells together." Adding a mosaic violation term produces a joint objective in natural units.
  • Deterministic: No random initialization, no resolution parameter, no community detection that can produce wildly different results across runs. The dendrogram is fully determined by the data.
  • Cut height = number of clusters: Choosing k clusters is just cutting the dendrogram at the appropriate height. No separate resolution sweep.
  • Proven quality on Gaussian blobs: Standard Ward on X should already match or exceed GMM for isotropic cases. We are adding a correction term, not replacing a good baseline.
  • Scipy and sklearn both implement Ward, making the modified version easy to build as a custom linkage matrix fed to scipy.cluster.hierarchy.fcluster.
Ward's one structural weakness for this problem: greedy, no backtracking. If an early merge is wrong (two cells from different mosaics fused because the spatial signal wasn't yet visible at the leaf level), Ward cannot undo it. The Leiden brainstorm's merge step attempted to address this with PAGA-guided post-hoc fixes. Ward will need an analogous correction mechanism.

2. Ward mechanics: the merge cost

Standard Ward linkage evaluates the variance increase caused by merging clusters A and B:

ΔW(A, B) = (n_A · n_B) / (n_A + n_B) · ‖μ_A − μ_B‖²

where n_A, n_B are cluster sizes, μ_A, μ_B are cluster centroids in feature space, and ‖·‖ is Euclidean distance. This is the squared Mahalanobis distance between centroids, scaled by the harmonic mean of cluster sizes. Properties:

  • The term grows with cluster size — merging large clusters is penalized more than merging leaves, which keeps clusters balanced.
  • ‖μ_A − μ_B‖² is always ≥ 0 and scale-dependent. Ward implicitly assumes feature dimensions are on comparable scales.
  • Ward satisfies the Lance–Williams recurrence, enabling O(N²) computation via the condensed distance matrix. The custom penalty may break this recurrence (see Section 8).

The standard algorithm maintains a priority queue of all current pairwise merge costs and repeatedly pops the minimum. With N=200 cells, this is a 200×200 matrix — trivially fast even with custom costs recomputed on the fly.

3. Adding the spatial penalty

The goal is to discourage merges that would place spatially overlapping cells in the same cluster. We add a penalty term that fires when the candidate merge pair (A, B) contains cell pairs that are spatial neighbors:

Δ_mosaic(A, B) = ΔW(A, B) + λ · V(A, B)

where V(A, B) is the mosaic violation score for the prospective merged cluster A∪B. The key design decision is how to define V.

3a. What V should measure

After merging A and B, cells from A and cells from B will share a cluster. The mosaic constraint says that spatial neighbors should belong to different clusters. So V(A, B) should count (or weight) the cross-cluster spatial neighbor pairs that would become same-cluster pairs after the merge:

V(A, B) = Σ_{i ∈ A, j ∈ B} ramp(score(i, j))

where the sum is over all pairs with one cell in A and one in B, and score(i, j) is a spatial proximity signal — either coverage fraction C(i, j) or the proximity score P(i, j) = 1 − d/r.

Only pairs within the spatial radius need to be considered; all others have score = 0 and contribute nothing. This keeps V sparse.

3b. Normalization

V as defined above grows with cluster size (more pairs to sum over). To make λ interpretable across different merge steps, normalize by the number of candidate cross-pairs:

V_norm(A, B) = (1 / n_A·n_B) · Σ_{i ∈ A, j ∈ B} ramp(score(i, j))

V_norm ∈ [0, 1]. It is the average spatial proximity between cells from A and cells from B, after the dead-zone ramp. This is directly comparable across merges of different sizes and makes λ dimensionless.

V_norm vs ΔW scaling ΔW is variance increase in feature units (squared). V_norm is in [0, 1]. λ therefore has units of "feature-space variance per unit of normalized mosaic violation." A useful starting rule: set λ so that a merge with V_norm = 1 (all cross-pairs are maximally overlapping) increases the effective merge cost by the same order as a typical ΔW at the leaf level. This provides an initial scale for λ calibration.

4. Dead zone and penalty ramp

Directly borrowed from the Leiden brainstorm (Section 2b of leiden_mosaic_constraints.html): the same τ_low / τ_high dead zone applies here.

ramp(c) = 0 for c ≤ τ_low = (c − τ_low) / (τ_high − τ_low) for τ_low < c < τ_high = 1 for c ≥ τ_high where c = coverage fraction C(i, j) or proximity P(i, j) τ_low = characteristic same-type contact level (e.g. ~0.10) τ_high = level where overlap is clearly cross-type (e.g. ~0.40)

The dead zone is more important here than in Leiden because Ward's greedy merge decisions cannot be undone. If same-type leaf pairs near the spatial boundary trigger small penalties at early merge steps, they will slightly inflate the cost of early correct merges, potentially diverting the algorithm to wrong merges. A well-calibrated τ_low prevents this.

4a. Adaptive τ_low for Ward

The same adaptive strategy used in the Leiden implementation applies: use the 75th percentile of all pairwise spatial scores within the spatial radius as a global τ_low estimate. This works well when the majority of within-radius pairs are same-type (which is expected given the mosaic structure — same-type cells tile the space, so within a given radius, they are the most common pairs).

τ_low is most critical at early merge steps. At leaf level, the "clusters" are singletons. V(A={i}, B={j}) = ramp(score(i, j)) for a single pair. If τ_low is set too low, any two spatially adjacent leaves with mild overlap will have elevated merge cost — including same-type leaves that should merge early and cheaply. This can fracture natural feature-based clusters before the algorithm gets to the interesting cross-type disambiguation.

5. Penalty variants

V1 — Hard gate

Refuse merges with any high-overlap cross-pair

If any pair (i ∈ A, j ∈ B) has score > τ_high, the merge cost is set to ∞ (or a very large sentinel). Ward will never merge A and B. All other merges proceed normally.

Pros

  • Cleanest possible enforcement — guaranteed no same-cluster high-overlap pairs
  • λ is irrelevant — only τ_high matters
  • Easy to debug: the dendrogram has explicit "forbidden" merges

Cons

  • May fragment well-separated clusters if τ_high is too low
  • Single outlier pairs can block otherwise correct merges
  • No graceful degradation — even one high-overlap pair blocks the whole merge
V2 — Soft additive

Add λ·V_norm to Ward cost

The standard approach: Δ_mosaic(A, B) = ΔW(A, B) + λ·V_norm(A, B). Mosaic violation increases the effective merge cost continuously. The algorithm still finds a valid dendrogram; it just prefers feature-similar, spatially non-overlapping merges.

Pros

  • Graceful: λ=0 recovers standard Ward; λ→∞ approaches hard gate
  • Differentiable in λ — can sweep or tune
  • V_norm normalization makes λ interpretable across merge sizes

Cons

  • Must choose λ — see Section 6
  • For very large λ, a heavily overlapping but featurally far pair could still be forced to merge early (if all other options are even worse)
  • Breaks the Lance–Williams recurrence — requires O(N²) recompute at each step
V3 — Multiplicative inflation

Scale Ward cost by (1 + λ·V_norm)

Instead of adding, multiply: Δ_mosaic(A, B) = ΔW(A, B) · (1 + λ·V_norm(A, B)). This keeps the penalty proportional to the base Ward cost, so it scales naturally with cluster size and feature separation.

Pros

  • Self-scaling: large ΔW merges get a large absolute penalty, leaf merges get a small one
  • λ=0 recovers Ward; no sudden jump at λ=1
  • No need to separately calibrate λ to feature units (V3 vs V2)

Cons

  • Penalty is small when ΔW is small — two featurally identical overlapping leaves get almost no penalty, which is exactly the hard case
  • Less intuitive than additive for choosing λ
  • Combines two sources of scale (ΔW and V_norm) in a nonlinear way
V4 — Penalty on merge, not on distance

Penalize at the cluster merge decision, not in feature space

Precompute all pairwise spatial scores. At each merge step, check only the new cross-pairs introduced (cells from A meeting cells from B for the first time). Penalty = sum of ramp values over new cross-pairs only. This avoids re-summing over within-A or within-B pairs that were already penalized at earlier merges.

Pros

  • Each pair (i, j) contributes to the penalty at most once — when their clusters first merge
  • Avoids double-counting same-pair overlaps at multiple merge steps
  • Theoretically cleaner: the penalty is a function of the merge, not of the current cluster state

Cons

  • No normalization by cluster size — penalty grows with n_A·n_B
  • Early merges produce small penalties (few cross-pairs); large merges produce large ones — may not fire at the right moment
  • Cannot be normalized the same way as V2/V3 without reintroducing size dependence

6. Choosing λ: dimensionless scaling

The central calibration challenge. ΔW is in squared feature units; V_norm ∈ [0, 1]. λ must bridge these. Two principled approaches:

6a. Data-driven λ: match typical leaf Ward cost

Compute the median ΔW cost across all leaf-level merges (i.e. merges at the bottom of the dendrogram where clusters are size 1 or 2). Set:

λ = α · median_ΔW_leaf

where α ∈ [0.5, 5] is a tunable scalar. This ensures that a merge with V_norm = 1 costs roughly α × median leaf merge. With α = 1, the spatial penalty is "as important as a typical feature-distance merge at the leaf level." With α = 5, spatial overlap dominates early decisions.

Why leaf-level calibration matters The penalty must be large enough to matter at the most critical moment — when two individual cells from different mosaics that happen to have similar features are candidates for merging. If λ is calibrated to late-stage (large cluster) merges, it will be too small to prevent early wrong merges between featurally similar cross-mosaic leaves.

6b. Fraction-of-cost cap

Alternatively, cap the penalty to at most f × ΔW for any single merge:

Δ_mosaic(A, B) = ΔW(A, B) · (1 + min(f, λ·V_norm(A, B) / ΔW(A, B)))

This prevents the penalty from completely dominating feature-based decisions when ΔW is very small (e.g., nearly identical feature vectors). With f = 2, the spatial penalty can at most triple the effective merge cost. Disadvantage: adds another hyperparameter.

6c. Sweep and validate with ARI

In practice: run the algorithm for λ ∈ {0, 0.1, 0.3, 1, 3, 10} × median_ΔW_leaf, cut the dendrogram at k clusters, measure ARI against ground truth. The plateau region (stable ARI across λ) is the sweet spot. A narrow plateau suggests the problem is genuinely hard; a wide one means the algorithm is robust.

Does a λ that works for anisotropic also work for moons/circles? Or does each dataset require separate tuning? Cross-config robustness is the key test of whether V_norm normalization is truly dimensionless.

7. Implementation sketch

The modified Ward algorithm does not fit scipy's built-in linkage — it uses a custom per-step cost function. The cleanest implementation is a direct greedy heap approach.

Precompute spatial scores

Build a spatial pair index: all (i, j) pairs within spatial_radius. For each pair, compute score(i, j) = coverage C(i, j) or proximity P(i, j). Store as a dict: (i, j) → raw_score and (i, j) → ramp(raw_score).

Initialize clusters

Each cell is its own cluster. Maintain a dict: cluster_id → set of cell indices. Precompute initial merge costs for all N(N-1)/2 pairs using Δ_mosaic(A={i}, B={j}).

Priority queue over merge costs

Use a min-heap on merge costs. At each step, pop the minimum-cost merge (A, B). Form cluster C = A ∪ B. Remove A and B from active clusters.

Recompute merge costs for new cluster C

For each remaining active cluster D, compute Δ_mosaic(C, D). The Ward term uses the Lance–Williams formula for ΔW. The penalty V_norm(C, D) requires iterating over all (i ∈ C, j ∈ D) pairs with nonzero ramp score — sparse due to spatial indexing. Push (C, D) cost onto the heap.

Repeat until k clusters remain

Record the merge sequence (linkage matrix) for later inspection. Cut at k clusters using scipy.cluster.hierarchy.fcluster on the custom linkage matrix, or simply stop when N − k merges have been made.

7a. Data structures

  • clusters: dict[int, set[int]] — active cluster id → member cell indices
  • cluster_mean: dict[int, np.ndarray] — running mean in feature space per cluster
  • cluster_size: dict[int, int] — running size
  • spatial_ramp: dict[tuple[int,int], float] — precomputed ramp(score(i,j)) for spatial pairs
  • heap: list[tuple[float, int, int]] — (cost, cluster_A, cluster_B); use lazy deletion for stale entries

7b. Computing V_norm efficiently

Naively, V_norm(C, D) requires iterating over |C| × |D| pairs. But the spatial score dict is very sparse — only pairs within spatial_radius are nonzero. Maintain a spatial adjacency index: cell_spatial_nbrs[i] = list of cell indices j where ramp(score(i, j)) > 0. Then:

V_norm(C, D) = (1 / |C|·|D|) · Σ_{i ∈ C} Σ_{j ∈ cell_spatial_nbrs[i] ∩ D} ramp(score(i, j))

Since the intersection with D is typically small (spatial neighbors are few), this is O(|C| × avg_spatial_degree) per merge step, not O(|C|×|D|).

8. The greedy problem: early mistakes persist

Ward is fully greedy — once two cells are merged, they remain merged. This is a fundamental limitation for mosaic-constrained clustering, and it manifests in two specific failure modes:

8a. Failure mode 1: featurally identical cross-mosaic leaves

Two cells from different mosaics with nearly identical feature vectors (X_i ≈ X_j) will have ΔW ≈ 0 even with the spatial penalty ≈ 0 (if they are not close spatially). They may merge early simply because no other merge is cheaper. Once merged, every subsequent merge involving this cluster carries the cost of having a mixed cluster, making it progressively harder to correct.

This is the same scenario that defeats GMM In the anisotropic test case, ARI = 0.931 for GMM because ~3 cells are ambiguous in feature space. Ward without spatial penalty would likely get similar ARI. The penalty only helps for cells that are both featurally ambiguous AND spatially adjacent — i.e., cells near mosaic boundaries. The spatial signal provides a tie-breaker, but only when it fires.

8b. Failure mode 2: correct early merges perturb later spatial costs

When two correct same-type cells merge at the leaf level, the resulting cluster now "covers" a larger region of space. Its spatial overlap with nearby clusters of different types increases, which correctly increases the penalty for later wrong merges. But it may also increase the penalty for later correct merges if the new cluster straddles a region with high feature-space similarity to another type.

8c. Mitigation: post-hoc reassignment

After the main Ward pass, inspect all mixed-cluster pairs (clusters that contain cells with high spatial overlap to a different cluster). Greedily reassign borderline cells to the cluster that minimizes Δ_mosaic evaluated locally. This is equivalent to the merge step in the Leiden brainstorm, but simpler because the cluster structure is already nearly correct.

Reassignment creates a chicken-and-egg problem. To know which cells are "borderline," you need to know the correct cluster assignment. To get the correct assignment, you need to handle borderline cells. One approach: iterate reassignment until convergence (analogous to k-means refinement). This adds a hyperparameter (max iterations) but is bounded because each reassignment strictly decreases total Δ_mosaic.

8d. Alternative: constrained complete-linkage post-pass

After Ward produces k clusters, run a second agglomerative pass using complete linkage but with a hard constraint: never merge two clusters if merging them would create any (i, j) pair with coverage fraction > τ_high. This is effectively a "mosaic-safety check" filter on the existing partition. It is cheap because it only operates on k clusters (small N), not on individual cells.

9. Open questions

Should we use coverage fraction C(i, j) or proximity P(i, j) = 1 − d/r as the spatial score? C fires only when polygons actually overlap (more biologically principled, but zero for most pairs). P fires for any pair within radius r (more signal, but also more false positives for cells that happen to be nearby but are the same type).
Is V2 (additive) or V3 (multiplicative) better in practice? V3 self-scales but is weak for featurally identical leaves. V2 is stronger at leaf level but requires λ calibrated in feature units.
How does Ward handle the non-convex cluster cases (moons, circles)? Standard Ward assumes ellipsoidal clusters. The penalty won't help if the base Ward cost function itself fails for non-convex geometry — we'd need a non-Euclidean dissimilarity (e.g., UMAP embedding + Ward).
Is Ward strictly better than GMM on the toy configs, without any spatial penalty? If Ward(λ=0) ≈ GMM in ARI, the spatial penalty adds value only for the ambiguous boundary cells. If Ward is strictly worse (e.g., for anisotropic clusters), we may need to run Ward in a projected space (PCA, whitening) before adding the penalty.
Can we break the Lance–Williams recurrence bottleneck? Standard Ward runs in O(N² log N) using the recurrence. Custom Δ_mosaic requires recomputing V_norm at each step. With N≈200 and sparse spatial pairs, O(N² + N × avg_degree) per step is feasible, but for N=2000 this may become slow. Is there a closed-form recurrence for V_norm(C∪D, E) from V_norm(C, E) and V_norm(D, E)?
What is the right way to evaluate success? ARI against ground truth is clean for toy data, but for real data we want to minimize mosaic violations (fraction of within-cluster spatial neighbor pairs) while maximizing feature compactness. These two objectives may conflict — what is the right Pareto tradeoff?
Should λ be annealed? Start high (enforce spatial constraint early, separate wrong-type neighbors) and decay (let feature similarity dominate at large cluster scale). This is analogous to simulated annealing but applied to a greedy algorithm — the "temperature" controls the relative weight of the two objectives at different merge levels.

10. Empirical findings: anisotropic benchmark

Implemented and tested on configs/anisotropic.yaml (3 mosaics, 208 cells, 2D Gaussian features with aspect_ratio=4, random rotation, overlap_factor=1.1). Standard Ward (lam=0) and modified Ward (lam=1) both achieve ARI=0.931 — identical to GMM. The penalty has zero effect across the full lam sweep (0 → 50).

10a. Why the penalty is ineffective: the spatial pivot problem

Five cells are misclassified. Tracing the Ward merge path reveals the structural failure:

Cell 76 (true=0, wrong=2): Nearest wrong-type cell with spatial proximity: nb=195, spat=5.90, prox=0.465, cost=0.00008 After penalty (lam_scaled=0.08): cost for nb=195 becomes 0.082 → algorithm pivots to: nb=187, spat=74.41, prox=0.000, cost=0.00023 (NO penalty) Best correct-type merge: nb=35, feat=0.037, spat=large, cost=0.00069 Result: still merges with wrong cluster, just via a different (farther) cell

The penalty makes the NEAREST wrong-type neighbor expensive, but there are dozens of wrong-type cells at large spatial distances with no penalty and still much smaller feature distances than any correct-type cell. The wrong cluster absorbs the misclassified cell regardless.

The spatial pivot: the penalty creates a cost re-routing problem For each misclassified cell, there are ~50-80 cells from the wrong type. At most 2-3 of them are within spatial_radius. Penalizing those 2-3 cells routes the algorithm to any of the remaining 50-75 unpenalized wrong-type cells. As long as the wrong cluster is "everywhere in feature space," spatially local penalties cannot box it out.

10b. Spatial neighborhoods reveal the correct signal — but not via proximity

Despite the penalty failure, the spatial neighborhood composition carries informative signal for 3 of the 5 misclassified cells:

Cell 76 (true=0): spat nbrs = {type0: 4, type1: 3, type2: 2} feat10 = {type0: 4, type2: 6} Cell 77 (true=0): spat nbrs = {type0: 3, type1: 2, type2: 1} feat10 = {type0: 2, type2: 8} Cell 94 (true=0): spat nbrs = {type0: 4, type1: 1, type2: 1} feat10 = {type0: 3, type2: 7}

For cells 76, 77, 94: the spatial neighborhood has a plurality of the correct type, but the feature neighborhood is dominated by the wrong type. The spatial signal points to the right answer — but through a different mechanism than pairwise proximity penalties. The signal is: "your spatial neighbors are mostly type 0, so you are probably type 0 too" (complementarity: if your neighbors are type 0, you can't be type 0 — wait, that's the opposite).

Actually: the mosaic prior says cells of the same type should NOT be spatial neighbors. So "4 of your 9 spatial neighbors are type 0" is NOT evidence that you are type 0 — it might be evidence that you are NOT type 0. But wait, these 4 type-0 spatial neighbors are the correct same-type cells forming the local type-0 mosaic tile. Cells 76/77/94 ARE type 0, and their type-0 spatial neighbors are their correct mosaic partners.

The mosaic signal is ambiguous for individual cells. "Many spatial neighbors of type A" could mean: (1) you are type A and this is your mosaic tile, or (2) you are surrounded by type A because you are a different type living in type-A territory. The mosaic constraint rules out (2) only in expectation — locally, at a single cell, either interpretation is valid. This is why the spatial penalty cannot be applied naively.

For cells 176, 187 (true=2): their spatial neighborhoods contain zero type-2 cells, and their feature vectors look like type 0. These are genuinely ambiguous — no spatial signal can correct them without global context about where the type-2 mosaic tile is supposed to be.

10c. Summary of failure mode taxonomy

  • Type A (cells 76, 77, 94): featurally ambiguous but spatially placed correctly. Feature space has wrong-type cells at every distance; spatial proximity penalty is defeated by routing. The spatial neighborhood composition is informative — a post-hoc MRF refinement over the final assignment (rather than the merge step) is the natural next approach.
  • Type B (cells 176, 187): genuinely ambiguous — wrong features AND wrong spatial context (zero same-type spatial neighbors). Not fixable with spatial information alone.

The next direction — treating cell type assignment as a Markov Random Field and refining after an initial GMM/Ward pass — is developed in mrf_mosaic_constraints.html.
