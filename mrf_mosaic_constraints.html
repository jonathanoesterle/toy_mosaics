1. Why MRF: what Leiden and Ward missed

Both Leiden and Ward attempted to enforce the mosaic constraint during clustering — by modifying edge weights or merge costs before committing to a partition. Both failed on the anisotropic benchmark (ARI stuck at 0.931 regardless of penalty strength). The root cause was the spatial pivot problem: penalizing the nearest spatially-close wrong-type candidate does not prevent merging with any of the many other wrong-type cells that are spatially distant and therefore unpenalized.

The wrong merges happen with spatially distant cells. Any penalty that only fires for spatially close pairs cannot box out a wrong cluster that is "everywhere in feature space."
The key shift: act on the final assignment, not on the merge path Instead of penalizing individual merge decisions, operate on the complete label assignment y = (y_1, ..., y_N) and penalize configurations where two spatially adjacent cells share the same label. This is evaluated globally, after all clustering decisions are made, and can inspect the full spatial neighborhood at once.

2. The energy formulation

Treat cell type assignment as MAP inference in a pairwise MRF defined over the spatial graph. The energy to minimize is:

E(y) = Σ_i ψ_i(y_i) + λ · Σ_{(i,j) ∈ S} w_ij · φ(y_i, y_j) y_i ∈ {0, 1, ..., K-1} cell type assignment ψ_i(y_i) unary cost: how well does X_i fit cluster y_i? φ(y_i, y_j) pairwise cost: do cells i and j have conflicting types? S spatial edge set: pairs within spatial_radius w_ij ∈ (0, 1] proximity weight or coverage fraction for pair (i, j) λ global penalty strength

Lower E is better. The unary term comes from the GMM; the pairwise term encodes the mosaic prior. Finding the exact minimum is NP-hard for K ≥ 3, but ICM converges reliably for small K and sparse graphs.

3. Unary term: GMM posteriors + freezing CURRENT

The unary cost ψ_i(y_i) quantifies how well cell i's feature vector X_i fits cluster y_i:

ψ_i(k) = -log P(y_i = k | X_i) = -log [ π_k · N(X_i; μ_k, Σ_k) / Σ_{k'} π_{k'} · N(X_i; μ_{k'}, Σ_{k'}) ]

where π_k, μ_k, Σ_k are the GMM mixture weight, mean, and covariance for component k. Cells with max-posterior P(y_i = y* | X_i) ≈ 1 are unambiguous; cells with spread posteriors (max ≈ 0.4–0.6 for K=3) are candidates for spatial correction.

Confidence-based freezing

Before ICM, high-confidence cells are frozen — their labels act as anchors and are never updated:

frozen(i) iff P(y_i = y*_i | X_i) >= conf_threshold

The vast majority of cells (~95%) are frozen. This bounds the damage radius: the MRF adjusts only a small fraction of cells, so an incorrect reassignment cannot globally destabilize an already-good GMM solution.

ALTERNATIVE

Ward-based unary terms

Ward does not produce posterior probabilities naturally. Three options exist if Ward (rather than GMM) were used as the initial clusterer: distance to centroid ψ_i(k) = ‖X_i − μ_k‖²; soft Ward (fit GMM initialized at Ward centroids); or Mahalanobis distance using per-cluster covariance. The Mahalanobis option is most principled and recovers the same posterior as GMM for anisotropic Gaussian clusters.

4. Pairwise term: Potts and signed-ramp CURRENT — BOTH VARIANTS

The pairwise term encodes the mosaic constraint: two spatially adjacent cells should have different types. Both variants below are implemented and compared per config in generate_mrf_figures.py.

CURRENT

Plain Potts model

The standard choice: any same-type adjacency incurs cost proportional to proximity weight w_ij.
φ_Potts(y_i, y_j) = [y_i == y_j] Full pairwise cost: w_ij · [y_i == y_j] w_ij = proximity weight in (0, 1] (higher = closer)

This is label diversification (the opposite of the smoothing prior used in image segmentation). Every same-type adjacent pair incurs a penalty, including cells in the same mosaic tile that happen to be close — this is the source of the anti-helpful problem for cell 77 (§8).

CURRENT

Signed-ramp variant

Changes the sign of the pairwise term for low-CF same-type contacts. Intra-tile adjacency (low coverage fraction) becomes attractive; territorial overlap (high CF) remains repulsive. Addresses the Potts anti-helpful problem for cell 77.
φ_signed(y_i, y_j, CF_ij) = [y_i == y_j] · signed_ramp(CF_ij) signed_ramp(CF) = −(τ_low − CF) / τ_low for CF < τ_low (attractive: same-tile contact) = 0 for CF = τ_low = +(CF − τ_low) / (τ_high − τ_low) for τ_low < CF < τ_high (repulsive: territorial overlap) = +1 for CF ≥ τ_high Current default: tau_low = 0.30 → covers intra-tile CF range 0.20–0.26, putting those contacts in the attractive zone → invoked via MRFMosaicStrategy(signed_ramp=True, tau_low=0.30)
ALTERNATIVE

Type-asymmetric penalties

If prior knowledge about per-type same-type contact rates is available, the dead zone can be made type-specific: φ_asym(y_i, y_j, w_ij) = [y_i == y_j] · max(0, w_ij − τ_low(y_i)), where τ_low(k) is the expected CF for same-tile contacts of type k. Not implemented.

5. Inference: ICM CURRENT

ICM (Iterated Conditional Modes) is the inference algorithm. It sweeps over all unfrozen cells repeatedly, updating each to its locally optimal label:

E_local(i, k) = ψ_i(k) + λ · Σ_{j ∈ spatial_nbrs(i)} w_ij · φ(k, y_j) y_i ← argmin_k E_local(i, k) for each unfrozen cell i, each sweep

Sweep until no cell changes label (convergence). Total E(y) decreases monotonically — guaranteed to converge in finite steps to a local minimum.

Initialize

Start from GMM labels y. Freeze high-confidence cells (posterior ≥ conf_threshold).

Cell sweep

For each unfrozen cell i, compute E_local(i, k) for every label k. Assign y_i = argmin_k E_local(i, k).

Repeat until convergence

Continue sweeping until no cell changes label. Record n_iters, n_changed, violations_before / violations_after in the model dict.

ICM considers all labels for every cell at every sweep, not just cells currently in violation. This means it can spontaneously correct cells with no current same-type conflict if a neighboring change opens up a better assignment. For K=3 and N≈200 with a sparse spatial graph, convergence is typically fast (2–5 sweeps).

ICM converges to a local minimum, not the global optimum. Starting from a good GMM solution (ARI ≥ 0.93), the local minimum is usually close to the global one. For configs with worse GMM initialization, the echo-chamber effect — wrong cells reinforcing each other — can trap ICM. The monotone convergence guarantee does not protect against this.

6. Lambda, spatial radius, and tau calibration

Lambda calibration

Lambda controls the trade-off between feature fidelity and spatial consistency. The natural scale: for a GMM unary term, the posterior gap for an ambiguous cell (K=3) is about log(3) ≈ 1.1 nats; the maximum pairwise term is w_ij · 1 ≤ 1. So λ ≈ 1 makes the penalty for a maximally-close same-type pair comparable to a 2.7× likelihood ratio.

Empirically, higher λ (e.g. λ=20) is used in the anisotropic analysis. The adaptive calibration below provides a principled alternative:

lambda_adaptive = alpha · median_posterior_gap posterior_gap(i) = P(y_i = 1st | X_i) - P(y_i = 2nd | X_i) median = over all cells (or over ambiguous cells only)
Too-large lambda: spatial coloring. If λ dominates, the algorithm tries to make every spatial neighbor pair have different labels — producing a graph coloring rather than a feature-based clustering. For K=3 in a dense mosaic, some same-type adjacent pairs are inevitable and correct. Lambda must allow this; the signed-ramp partly addresses it by making low-CF contacts attractive.

Spatial radius

The spatial graph is built from all pairs within spatial_radius. The auto-computed default:

spatial_radius = 3.0 × median(nearest-neighbour distances) → typically ~11 units for the standard configs

This is set in generate_mrf_figures.py from KDTree(dataset.centers).query(..., k=2). Increasing the radius is a possible fix for spatially isolated cells like cell 94 (§8b) but trades specificity for coverage.

Inferring tau_low and tau_high from data

Both thresholds are currently set by hand. In a real dataset (no ground truth) they need to be inferred from the pair CF distribution itself. The starting point is to understand what that distribution looks like:

Empirical CF distributions on anisotropic.yaml (within spatial_radius ≈ 11): same-type pairs (n=121): p50=0.079 p95=0.177 max=0.260 diff-type pairs (n=362): p5 =0.109 p50=0.492 max=1.000 Overlap zone ≈ [0.11, 0.26] — pairs in this range could be either type.

The distributions are not cleanly bimodal with a gap. The overlap region is genuine: a same-tile Voronoi contact and a mild cross-type territorial touch can produce the same CF. This rules out a perfect threshold and means any choice of tau_low involves a trade-off between capturing all same-tile contacts (high tau_low, includes some diff-type) and avoiding false co-type signals (low tau_low, misses some same-tile contacts).

OPTION A

KDE valley detection on all pair CFs

Fit a kernel density estimate to the CF values of all pairs within spatial_radius. A well-structured mosaic produces a bimodal distribution: a low-CF peak (same-tile contacts) and a high-CF peak (territorial overlaps). tau_low = first local minimum after the low-CF peak; tau_high = peak of the high-CF mode.

from scipy.signal import argrelmin from scipy.stats import gaussian_kde kde = gaussian_kde(all_cfs, bw_method=0.15) x = np.linspace(0, 1, 500) minima = argrelmin(kde(x), order=10)[0] tau_low = x[minima[0]] # first valley tau_high = x[kde(x).argmax() ... ] # peak of upper mode

Pros

  • Fully unsupervised — no labels of any kind required
  • Natural: the valley is the maximum-uncertainty point of the distribution
  • tau_high falls out automatically from the same fit

Cons

  • Bandwidth choice is critical — too wide smooths out the valley; too narrow gives spurious minima
  • Fails for high_overlap config: same-type max CF = 0.60, diff-type p5 = 0.33 — the two modes overlap heavily, the valley may not exist or may be mislocated
  • KDE valley is sensitive to the ratio of same-type to diff-type pairs (which depends on K and density)
OPTION B

Frozen-anchor calibration

Use the GMM's own high-confidence cells as a proxy for ground-truth labels. Cells with max-posterior ≥ conf_threshold are already identified as frozen for ICM. Their pairwise CF distribution closely mirrors the true same/diff-type CF distribution — because frozen cells are assigned correctly with high probability.

Frozen-cell calibration results on anisotropic.yaml: Frozen cells: 190 / 208 (91%) Frozen same-label pairs (n=109): p95=0.176 max=0.246 Frozen diff-label pairs (n=300): p5 =0.115 True same-type: p95=0.177 max=0.260 (cf. frozen same above) True diff-type: p5 =0.109 (cf. frozen diff above) tau_low = max(frozen same-label CF) ≈ 0.25 (or p99 ≈ 0.19 for a conservative estimate) tau_high = median(frozen diff-label CF)

Pros

  • Uses information already computed for ICM — no extra work
  • Adapts per-dataset: different cell densities and K values produce different CF ranges, all captured automatically
  • Very accurate: frozen same-label max CF (0.246) matches true max (0.260) closely
  • Natural bootstrap: run GMM → calibrate tau → run MRF → re-calibrate if needed

Cons

  • Circular if GMM labels are wrong: a systematic GMM error (wrong cluster split) will produce wrong tau estimates
  • Fails in the limit of few frozen cells (rare when conf_threshold = 0.90 gives 91% frozen)
  • Does not detect same-type pairs among the unfrozen cells — but those are ambiguous anyway
OPTION C

Per-cell minimum CF

For each cell i, the pair with the lowest CF among all its spatial neighbors is almost certainly a same-tile contact (the most adjacent, least overlapping neighbor). The distribution of per-cell minimum CFs therefore approximates the same-tile CF distribution without needing any labels.

min_cf[i] = min(CF(i, j) for j in spatial_nbrs(i)) tau_low = np.percentile(min_cf, 95) # captures 95% of same-tile minimum contacts

Pros

  • Completely label-free
  • Concentrates on the most same-tile-like contacts by construction
  • Robust to outliers: takes a percentile over all cells

Cons

  • The minimum CF of a cell's neighborhood is not guaranteed to be a same-tile contact — for clipped or isolated cells, it may be a cross-type contact with anomalously low CF
  • Biased downward when cells have many neighbors (more chances for a very-low-CF pair)
  • Does not directly help with tau_high
OPTION D

Geometric estimate from cell area and NND

For a mosaic of density ρ (cells per unit area), cells from the same type have mean territory area ≈ K/ρ. Two adjacent same-tile cells (NND ≈ 1/√(ρ/K)) have Voronoi cells that partition space without overlap — but the convex-hull approximation adds spurious overlap at concave boundaries. The expected same-tile CF is therefore a function of cell geometry (aspect ratio, regularity) and can be estimated analytically or from a simulated calibration run.

density = N / domain_area mean_territory_area = K / density expected_same_tile_NND = 1 / sqrt(density / K) # Expected convex-hull CF from geometry: 0.05 to 0.15 for typical Voronoi cells # tau_low = estimated_max_same_tile_CF * safety_margin (e.g. 1.5×)

Pros

  • Purely physics-based — works even with very few cells or extreme configs
  • Provides a prior that can be combined with data-driven estimates

Cons

  • Requires knowing K (number of types) — which is what we're trying to estimate
  • Requires knowing domain area — not always well-defined for partial reconstructions
  • The safety margin is empirical; the convex-hull CF depends on cell shape irregularity in hard-to-predict ways
OPTION E

Grid search with unsupervised mosaic quality score

Sweep tau_low over a grid (e.g. 0.05 to 0.50 in steps of 0.05) and for each value run the signed-ramp MRF. Evaluate each result using an unsupervised quality score that does not require ground truth:

  • Violation rate: fraction of active-repulsive edges (CF > tau_low) that connect same-label cells. Lower is better — but can trivially decrease by making everything attractive (high tau_low).
  • NND regularity (CV): for each cluster, compute the coefficient of variation of nearest-neighbor distances within the cluster. Lower CV = more regular mosaic. Combine across clusters.
  • Feature compactness: intra-cluster feature variance. Should not increase as tau_low increases — if it does, the spatial term is overriding genuine feature structure.

The optimal tau_low jointly minimizes violation rate and NND-CV while not increasing feature compactness. This is expensive (one MRF run per candidate) but fully unsupervised.

Pros

  • Directly optimizes what we care about
  • Can discover a tau_low that works for this specific dataset

Cons

  • Expensive: 10 MRF runs per dataset
  • Violation rate can be gamed (very high tau_low makes all contacts attractive, reducing violations)
  • NND-CV can degrade non-monotonically — the optimum may not be well-defined
Recommended starting point: frozen-anchor calibration (Option B) The GMM posteriors are already computed before ICM runs. Extracting tau_low and tau_high from frozen-pair CF statistics adds ~5 lines of code and no additional computation. The frozen same-label CF distribution matches the true same-type CF distribution closely on toy data (max error < 0.02). It adapts automatically to different mosaic geometries, K values, and spatial scales without any additional assumptions. Proposed defaults derived from frozen pairs:
tau_low = np.percentile(frozen_same_cf, 99) # conservative: misses CFs in (p99, max) or max(frozen_same_cf) # aggressive: puts ALL frozen same-tile in attractive zone tau_high = np.percentile(frozen_diff_cf, 25) # 75% of confirmed cross-type pairs are at full repulsion

7. Diagnostics: plot_mrf_diagnostics IMPLEMENTED

plot_mrf_diagnostics(dataset, result, *, ground_truth=None) returns a 6-panel figure. Each config in generate_mrf_figures.py generates one diagnostics page for the plain MRF and one for the signed-ramp variant.

The six panels cover:

  • Conflict–confidence scatter — one point per cell; x = GMM max-posterior (confidence), y = total same-type pairwise weight (conflict). Color = correct/error if ground truth provided. Four quadrants identify fixable (ambiguous + conflicted), frozen-violated, Type-B no-signal, and safe cells.
  • Spatial energy-gap map — polygons colored by E_local(i, 2nd-best) − E_local(i, best). Large gap = stable; gap near zero = about to flip.
  • Neighbourhood composition bars — for each misclassified cell, a stacked bar of neighbor type composition before and after ICM. Requires ground truth.
  • Feature-space scatter with conflict overlay — GMM decision boundaries + per-cell conflict score coloring.
  • ICM trajectory arrows — cells that changed label during ICM, shown as arrows in feature space with energy gain. Requires per-cell trajectory logging in the model dict (not yet stored).
Per-cell ICM trajectories are not yet logged. The model dict stores n_changed (aggregate) and violations_before/after, but not the per-cell label sequence. The trajectory-arrow panel is therefore currently empty or skipped. To enable it, extend MRFMosaicStrategy.fit() to append (iteration, cell_i, old_label, new_label) per flip — zero algorithmic cost.

8. Known failure modes: cells 77 and 94

After ICM, two cells remain misclassified on anisotropic.yaml (spatial_radius≈11, λ=20). Both are true type 0 but predicted type 2. They represent two distinct structural failure modes.

8a. Cell 77 — the anti-helpful pairwise problem

Cell 77 has four spatial neighbours within the radius. All four are correctly assigned. The feature model gives ~67% posterior for the wrong type (type 2).

Cell 77 spatial neighbours (after ICM): cell 123: CF=0.561 w=1.000 label=type1 (correct) cell 127: CF=0.533 w=1.000 label=type1 (correct) cell 75: CF=0.260 w=0.533 label=type0 (correct) ← same-tile territory cell 83: CF=0.202 w=0.340 label=type0 (correct) ← same-tile territory ICM energy for each candidate label (plain Potts): type2 (current wrong): unary_2 + λ · 0 = baseline (no same-type nbrs) type0 (correct): unary_0 + λ · (0.533+0.34) ≈ unary_0 + 17.5 ← PENALTY for correct assignment type1: unary_1 + λ · (1.0+1.0) ≈ unary_1 + 40 ICM chooses type2: feature model prefers it AND correct-type neighbours add penalty.
The Potts pairwise term is anti-helpful for cell 77. Cells 75 and 83 — correctly assigned type-0 neighbours — create a penalty for assigning cell 77 to type 0. From ICM's perspective, type-0 neighbours are evidence against becoming type 0. The correct interpretation — "I am in type-0 territory" — is the opposite of what the Potts model computes.

The signed-ramp variant partially addresses this: cells 75 (CF=0.26) and 83 (CF=0.20) are below or near tau_low=0.30, so their contribution to E_local(77, type0) becomes negative or near-zero instead of +17.5. Whether this is sufficient to overcome the 67% feature preference for type 2 depends on the calibration of λ and α.

8b. Cell 94 — spatial isolation

Cell 94 sits at the domain edge (y≈4.5) with zero spatial neighbours within the auto-computed radius.

Cell 94: pos=[41.05, 4.54] true=type0 pred=type2 Spatial neighbours within radius 11: none ICM: E_local(94, k) = unary[94, k] for all k → GMM label unchanged Feature posterior: ~61% for type2 (wrong) Broader neighbourhood (ground-truth labels, no CF filtering): within r=20: type0=8 type1=5 type2=6 → type0 plurality within r=30: type0=14 type1=12 type2=10 → type0 plurality within r=50: type0=43 type1=27 type2=18 → type0 clear majority

The territorial signal exists at larger radii but the spatial graph has no edges for this cell, so none of it enters the energy function. ICM is fully blind to it.

9. Ceiling analysis: what is fixable

From the empirical results on anisotropic.yaml, errors split into two categories:

Fixable (Type-A): cell 76 — now corrected

Cell 76 (GMM: type 2, true: type 0) had same-type spatial conflicts after GMM assignment: being labelled type-2 placed it adjacent to other type-2 cells. ICM detected the conflict and successfully reassigned it. This is the MRF sweet spot: ambiguous feature signal + direct spatial conflict.

Persistent errors: cells 77 and 94

Cell 77 satisfies the conflict condition after GMM assignment but the pairwise term works against correction (§8a). Cell 94 has no edges in the spatial graph and is therefore invisible to ICM (§8b). Neither is a classic Type-B (no-conflict, no-signal) error; both are structural limitations of the current formulation.

Necessary conditions for ICM correction

For ICM to correct a wrong assignment y_i = wrong, cell i must:

  1. Have at least one spatial edge — i.e., ≥1 neighbour within spatial_radius.
  2. Have a posterior gap small enough that the spatial term (λ × pairwise) can overcome the unary preference for the wrong label.
  3. Not be surrounded by other wrongly-assigned cells that would reinforce the wrong label via the pairwise term.
  4. Have the pairwise term point in the correct direction — which the Potts model does not guarantee when the cell is embedded in correct-type territory.
Maximum achievable ARI improvement from the current formulation Cell 76 was fixed (ARI improved from ~0.93 to ~0.95). Cells 77 and 94 require either the signed-ramp (for 77) or a larger radius (for 94) — see §10 for these extensions. The 2 remaining errors represent the practical floor for the current spatial-graph formulation at radius 11.

10. Extensions and alternatives

EXTENSION

Larger spatial radius for edge-of-domain cells

Increasing spatial_radius from ~11 to ~20 would give cell 94 approximately 19 neighbours, making it visible to ICM. The type-0 plurality within r=20 (8 vs 5+6) could then provide a corrective pull — but only if the anti-helpful problem (cell 77) is also addressed.

Tradeoff: a larger radius captures more intra-tile same-type contacts, increasing false "violations" that the dead zone must absorb. The auto-computed radius (3× median NN distance) is already principled; inflating it is a targeted patch for edge cells rather than a general improvement.

ALTERNATIVE

Loopy Belief Propagation (LBP)

LBP propagates probability distributions through the spatial graph rather than hard label assignments, producing approximate marginal posteriors P(y_i = k | X, neighbours). Each cell sends messages to neighbours:

m_{i→j}(k) ∝ Σ_{k'} exp(-ψ_i(k')) · exp(-λ · w_ij · φ(k', k)) · Π_{l ∈ nbrs(i)\{j}} m_{l→i}(k') MAP assignment after convergence: y_i = argmax_k b_i(k) where b_i(k) ∝ exp(-ψ_i(k)) · Π_{j ∈ nbrs(i)} m_{j→i}(k)

Advantages over ICM

  • Propagates spatial context beyond 1-hop (useful for long-chain configurations)
  • Produces calibrated posterior probabilities, not just MAP
  • Can escape some local minima that trap ICM

Drawbacks

  • O(N · K² · deg · iters) vs O(N · K · deg · iters) for ICM
  • Convergence not guaranteed on loopy graphs
  • Overkill for K≤6, N≤300, sparse local graphs — ICM is likely equivalent in practice

11. Open questions

Does the signed-ramp (tau_low=0.30) actually fix cell 77 on anisotropic? Cells 75 and 83 have CF=0.26 and 0.20, which are just below tau_low=0.30 — they should be in the mild-attractive zone. Whether the resulting attraction is large enough to overcome the 67% feature preference for type 2 needs to be read from the generated figures.
What is the right lambda? The posterior-gap calibration (§6) gives a principled starting point, but λ=20 is used empirically. Is there a closed-form derivation that avoids per-config tuning?
Can the KDE prior (§10) fix cell 94 without introducing new errors near mosaic boundaries? The risk is that boundary cells near a type-0 region get incorrectly pulled to type 0 even when their feature signal says otherwise. A precision/recall analysis of the KDE augmentation across all configs is needed before committing.
Is ICM sufficient, or do we need LBP? For the current configs (K≤3, N≤300, sparse local graph), ICM likely reaches the same local minimum as LBP. The question becomes relevant only if we add configs with longer spatial chains or weaker GMM initialization.
Does the KDE prior + signed ramp combination fix both remaining cells when used together? Each alone may be insufficient for cell 77 (67% feature gap is large). The combination reduces unary preference for type 2 (KDE) and converts the pairwise penalty into an attraction (signed ramp). Whether together they cross the threshold is cheap to test.
Can the MRF be used unsupervised for real data (no ground truth)? The frozen-anchor approach (§6) calibrates tau_low and tau_high from GMM posteriors without requiring external labels. The remaining open question is whether the GMM labels of frozen cells are reliable enough to serve as anchors for tau calibration when the initial clustering is worse than on toy data (ARI < 0.85).

12. Test suite design

Tests should cover three layers: smoke (shape/dtype/reproducibility), quality (ARI thresholds, violation monotonicity), and diagnostic (per-cell detail on failures).

Smoke tests

Extend the existing STRATEGIES parametrization in tests/test_clustering.py to include MRFMosaicStrategy. Verify labels shape/dtype, n_unique_labels == n_mosaics, and model dict keys: gmm, lam, n_iters, n_frozen, n_changed, violations_before, violations_after. Easy configs (example) are sufficient here.

Quality tests

For hard configs (anisotropic, elongated_clusters, high_overlap):

  • Non-regression: ARI(MRF) ≥ ARI(GMM) − 0.01. MRF must not hurt relative to its own initialization.
  • Monotone violations: violations_after ≤ violations_before. ICM is monotone by construction; this catches sign-flip bugs in λ.
  • Frozen cells unchanged: no cell with GMM max-posterior ≥ conf_threshold should change label. Guards against a frozen-mask bug.

13. Exponential repulsion at high CF

The current signed-ramp saturates at +1 for CF ≥ τ_high, treating a 60 % overlap and a 100 % overlap identically. For a near-complete territorial overlap (CF → 1), that saturation under-penalises the most egregious violations and lets ICM keep a wrong label when the unary term marginally favours it.

Proposed replacement: log-ramp (exponential in the penalty)

signed_exp_ramp(CF) = −(τ_low − CF) / τ_low for CF < τ_low (attraction, unchanged) 0 for CF = τ_low log(1 + α · (CF − τ_low)) ───────────────────────── for τ_low < CF ≤ 1 (super-linear repulsion) log(1 + α · (1 − τ_low)) α controls curvature: α = 1 → nearly linear (≈ current ramp) α = 10 → strong upward bend above τ_low α = 50 → near-step: flat until ~0.6 CF, then sharp rise

Alternatively, the raw penalty can be set directly to exp(β · (CF − τ_low)) − 1, renormalised to [0, 1]. At CF=0.9 with β=5 this is ~90× higher than at CF=0.5, compared to 2× for the linear ramp.

Expected effects

  • Stronger force on near-complete overlaps: a cell with a CF=0.9 neighbour gets a pairwise cost close to λ regardless of unary preference — effectively forcing a type-split for the most territorial violations.
  • Sharper energy landscape: ambiguous cells near the decision boundary flip more decisively; ICM may converge in fewer sweeps.
  • Self-amplifying correction: once a few cells near a high-CF pair are reassigned, their new labels reduce further conflicts, creating a cascade. For well-structured mosaics this is good; it accelerates clean-up.

How it can fail

Lambda calibration breaks. The current λ=20 is tuned against a pairwise term bounded in [−1, 1]. With an exponential ramp that exceeds 1 at high CF, the effective penalty for a CF=0.95 pair is λ × penalty > λ. The unary term (scaled in nats) no longer sets the natural energy unit for the spatial term. Re-calibration of λ is mandatory; the adaptive formula λ = α × median_posterior_gap must include the maximum possible ramp value in the denominator.
Runaway cascade on dense mosaics. In high-density configs (high_overlap.yaml), same-type pairs routinely reach CF=0.4–0.6 even without a real territorial conflict. An exponential ramp dramatically amplifies these moderate CFs, potentially flipping cells that were correctly assigned. The dead zone (CF < τ_low) must be wide enough to absorb all such contacts — otherwise the algorithm over-corrects into a graph-coloring mode where no two adjacent cells can share a type.
Numerical sensitivity near CF = 1. Convex-hull CF values of 1.0 are common when one cell's hull contains another entirely (e.g., a small edge cell). The derivative of the penalty function diverges if β is large. A hard cap (clip CF to 0.99 before applying the ramp) or a soft-max form prevents NaN/Inf penalties.
Asymmetric effect by cell size. Coverage fraction is not symmetric: CF(i→j) ≠ CF(j→i) when cells differ in size. The current implementation uses max(CF_ij, CF_ji) or the directed value depending on convention. Under exponential repulsion, whichever convention is used has a much larger effect: the bigger cell will always be penalised less (CF is smaller relative to its own hull), so the smaller cell bears most of the corrective force. This size-driven asymmetry can produce biologically implausible reassignments for boundary cells whose polygons are artificially large.
VARIANT

Piecewise exponential: linear below τ_mid, exponential above

Keep the linear ramp between τ_low and τ_mid (e.g. 0.6) to preserve current ICM behaviour for moderate overlaps, and switch to exponential only for CF > τ_mid. This localises the change to the most egregious cases and avoids recalibrating λ for the bulk of the CF range. τ_mid adds one more threshold to calibrate, but it can be fixed at a hard-coded 0.6 as an intermediate saturation point.
Recommendation: the log-ramp with α=10, clipped at CF=0.99, is the safest option. It increases the penalty at CF=0.9 by ~3.5× relative to the linear ramp without diverging, and the existing τ_low/τ_high calibration logic is still valid. λ should be re-calibrated by dividing by the new ramp maximum (log(1+α·(1−τ_low))/log(2)) before running ICM.

13b. CF > 0.9 as a near-certain labelling error

The framing above treats CF > 0.9 as a point on a continuous penalty curve — stronger repulsion, but still a soft signal. The user's observation forces a sharper reframing:

In a mosaic, cells of the same type tile space without territorial overlap. Two same-type cells with CF = 0.9 share 90 % of their convex-hull area — geometrically impossible unless one cell is almost entirely contained inside the other's polygon. This cannot occur in any legitimate same-type adjacency. CF > 0.9 between two same-label cells is therefore not a "strong violation" but a near-certain labelling error.

This matters because soft penalties — including the log-ramp — can still be overridden by the unary term if the unary gap is large enough. For a near-certain error, that override should not be possible.

The frozen-cell blind spot

There is a deeper problem: the current architecture makes CF > 0.9 violations systematically invisible when one or both cells are frozen.

Frozen cells: posterior ≥ conf_threshold (0.90) → never updated by ICM. Log-ramp penalty for CF=0.95, λ=20: 20 × 0.96 ≈ 19.2 nats Unary gap for a frozen cell (posterior=0.92): ψ(2nd) − ψ(1st) ≈ log(0.92/0.04) ≈ 3.1 nats (K=3) → The pairwise signal (19.2 nats) vastly exceeds the unary gap (3.1 nats). → ICM WOULD flip this cell — if it were not frozen. → But it IS frozen. The penalty is never evaluated. The violation persists.

This means the log-ramp is irrelevant for the worst violations: any pair where both cells are high-confidence (and therefore frozen) cannot be corrected by ICM regardless of how large λ × ramp(CF) becomes.

High-confidence cells can still be wrong — and CF > 0.9 proves it. A cell with GMM posterior = 0.92 is frozen. If it has a same-label neighbour with CF = 0.95, one of the two cells is assigned to the wrong type with near-certainty — yet ICM cannot correct either, because both may be frozen. Confidence is a proxy for correctness, not a guarantee. The spatial signal at very high CF is more reliable than the feature signal; the frozen mechanism inverts this hierarchy.

13c. Options for enforcing CF > θ_hard as a hard constraint

Let θ_hard ∈ [0.80, 0.90] be a threshold above which same-label adjacency is treated as a near-certain error rather than a soft violation. Three architectural options:

OPTION H1

Force-unfreeze cells in high-CF violations

Before ICM runs, scan all same-label pairs with CF > θ_hard. For each such pair, unfreeze the cell with the lower GMM max-posterior (the less confident of the two). The unfrozen cell is then eligible for ICM correction, and with λ × ramp(CF) >> unary gap it will almost certainly flip to the correct label.

for (i, j), cf in raw_map.items(): if cf > θ_hard and labels[i] == labels[j]: # unfreeze the less-confident cell less_conf = i if posteriors[i].max() < posteriors[j].max() else j frozen[less_conf] = False

Pros

  • Minimal change: one pre-pass, then ICM proceeds normally
  • Targeted: only unfreezes cells that are in proven violations — the vast majority of frozen cells are untouched
  • Self-consistent: unfreezing feeds back into ICM's neighbourhood updates, propagating the correction

Cons

  • Unfreezing the wrong cell: if the lower-confidence cell is actually correct and the higher-confidence cell is wrong, the unfreeze picks the wrong target. (Rare if the more-confident cell is truly correct, but possible in pathological configs.)
  • Chained violations: if A–B and B–C are both high-CF violations with the same label, unfreezing B (lower-confidence in A–B) may still leave the B–C violation unaddressed.
  • θ_hard needs to be set conservatively (≥ 0.85) to avoid unfreezing legitimate same-type edge contacts in high-overlap configs.
OPTION H2

Pre-pass greedy correction before ICM

Before ICM (and before considering the pairwise term at all), scan all pairs with CF > θ_hard and same label. For each, reassign the lower-confidence cell to the label with the best unary score that is different from the current label — ignoring the pairwise term entirely. This pre-pass resolves the most obvious errors before ICM inherits them.

for (i, j), cf in sorted_by_cf_desc: # highest CF first if cf <= θ_hard or labels[i] != labels[j]: continue less_conf = i if posteriors[i].max() < posteriors[j].max() else j # best alternative label (unary only) alt_labels = [k for k in range(K) if k != labels[less_conf]] labels[less_conf] = alt_labels[np.argmin(unary[less_conf, alt_labels])]

Pros

  • Works even if both cells are frozen — it operates before the frozen mask is applied to ICM
  • Processes in descending CF order, so the most egregious violations are corrected first
  • The corrected label is then the starting point for ICM, which refines it further

Cons

  • Uses only the unary term to pick the alternative label — does not consider the pairwise context. A cell reassigned in the pre-pass may immediately create new violations that ICM must later clean up.
  • The greedy order matters: correcting pair (A, B) first may cascade and affect pair (B, C). The final result depends on which violations were processed first.
OPTION H3

Two-tier penalty: soft below θ_hard, hard above

Set pairwise weights for CF > θ_hard to a fixed large value M, regardless of the ramp. Unfreeze all cells involved in CF > θ_hard violations (since the penalty is now irresistible, restricting them to their current label would just produce a stuck violation). ICM then resolves these as forced flips.

θ_hard = 0.85 M = max_unary_gap + Δ # guaranteed to dominate any unary preference w_ij = M if cf > θ_hard = signed_log_ramp(cf) otherwise

M can be estimated as the 99th-percentile unary gap across all cells, so the hard constraint dominates for every cell in the dataset without requiring per-cell calibration.

Pros

  • Principled: the hard-constraint region is explicitly separated from the soft-penalty region with a clear energy hierarchy
  • Unambiguous: CF > θ_hard same-label pairs will always be corrected by ICM (the energy to change is lower than the energy to stay)

Cons

  • Requires estimating M per dataset — a poorly calibrated M that is not large enough may still allow the violation to persist
  • Combining M with the soft log-ramp in the same energy function creates a discontinuity at θ_hard that can produce oscillations in ICM if multiple cells are simultaneously at or near the boundary
  • Must unfreeze cells in hard-constraint violations, which partially undermines the confidence-based freezing logic
OPTION H4

Post-ICM cascade correction

After ICM converges, scan for remaining pairs with CF > θ_hard and same label. For each, force-flip the lower-confidence cell (ignoring the frozen mask) to the best unary alternative. Re-run ICM from the corrected state. Repeat until no CF > θ_hard violations remain or a maximum number of post-passes is reached.

Pros

  • Completely decoupled from ICM — the core algorithm is unchanged
  • Each post-pass is cheap (scan + flip + short ICM convergence from a nearly-converged state)
  • Natural termination criterion: stop when no high-CF violations remain

Cons

  • Not monotone: the post-pass flips can increase the total energy (since we are overriding the frozen mask). Total energy may oscillate between passes.
  • Risk of infinite loop if two cells alternately flip each other. Cap at max_post_passes=3.

14. Cluster merging and splitting

The current pipeline assumes K (number of cell types) is known in advance. In practice K must be estimated, and any fixed estimate will be wrong — the more natural approach is to overcluster aggressively (pick K_over >> K_true) and then merge sub-clusters guided jointly by feature similarity and mosaic compatibility.

The core problem

Under-clustering is irreversible; over-clustering is fixable. If two true types are merged into one cluster from the start, no amount of spatial refinement can split them apart — the unary term has no signal for the split. Over-clustering produces many small, locally coherent clusters that span subsets of the true territory. These can always be merged; they cannot be split further without re-running the GMM.

Option A: overclustering + mosaic-driven hierarchical merging (recommended)

OPTION A

Overclustering + hierarchical merge

Run GMM with K_over = c · K_true (e.g. c=3–5). Each true mosaic is now represented by several sub-clusters. Merge sub-clusters greedily using a mosaic compatibility criterion:

merge_score(A, B) = feature_distance(A, B) × territorial_overlap(A, B) feature_distance = Mahalanobis distance between sub-cluster means (GMM-derived) territorial_overlap = fraction of sub-cluster A's spatial territory that overlaps sub-cluster B = mean CF between A-cells and B-cells (directed) Merge A and B iff: feature_distance(A, B) < δ_feat (clusters are feature-similar) AND territorial_overlap(A, B) < ε_terr (clusters do NOT share territory) AND merged_cluster passes mosaic regularity check (NND-CV below threshold)

Repeat until no pair satisfies the merge criterion. The result is K_final ≤ K_over clusters whose internal spatial structure is mosaic-regular and whose inter-cluster boundaries align with territorial borders.

Pros

  • Monotone: each merge reduces K, so the algorithm terminates in at most K_over steps
  • Combines both signals: feature proximity ensures biological coherence; territorial non-overlap ensures mosaic structure
  • Gracefully discovers K: stop when no further merges satisfy both criteria; final K_final ≈ K_true without knowing K_true
  • Sub-clusters from overclustering are small and locally coherent — each is a reliable GMM anchor for tau calibration

Cons

  • Greedy merging is order-dependent; early merges can block later beneficial ones. Use a priority queue (lowest merge_score first) to mitigate
  • The two criteria (feature + territorial) may conflict: two sub-clusters could be feature-close but spatially interleaved. A weighted scalarisation territorial_overlap + w · feature_distance introduces a free parameter
  • K_over must be large enough that each true type is split into ≥2 sub-clusters — otherwise a true type is never a merge candidate and K_final stays too large
  • ICM with K_over components is slower (K² pairwise terms per cell per sweep)
OPTION B

Spectral splitting of spatially incoherent clusters

After the initial clustering, identify clusters that are spatially fragmented — their cells do not form a single connected component in the spatial graph. Split each such cluster along the spatial graph cut with minimum spatial weight (normalised cut on the subgraph of that cluster). Each fragment becomes a new sub-cluster. Then run MRF on the expanded label set and re-merge post-ICM.

Pros

  • Directly targets the failure mode: a spatially fragmented cluster is a sure sign of a wrong merge in feature space
  • The spatial graph is already built for ICM — no extra data structure needed

Cons

  • A correctly assigned type can also be spatially fragmented if there are missing cells or a narrow corridor between territories — splitting it is wrong
  • Ncut requires a connected subgraph; isolated edge cells create singletons that complicate the split
  • The criterion "spatially fragmented" needs a threshold on minimum-cut weight that is config-dependent
OPTION C

BIC/AIC model selection with adaptive K

Run GMM for K = 2, 3, ..., K_max. For each K, compute a penalised score that combines GMM log-likelihood (feature fit) with a spatial regularity score (mean NND-CV across clusters, violation rate after MRF). Select the K that jointly minimises both. This is expensive (K_max MRF runs) but principled — it directly optimises the stated goal.

score(K) = −GMM_BIC(K) + γ · mosaic_violation_rate(K) + δ · mean_NND_CV(K) γ, δ trade off feature fit vs spatial structure quality

Pros

  • Directly answers "what is K?" without any over-clustering heuristic
  • The spatial regularity term makes it sensitive to mosaic structure, not just feature density

Cons

  • Very expensive: K_max full MRF runs including coverage map construction
  • The spatial regularity score is not smooth in K — small changes in K can cause large changes in ICM convergence points
  • γ and δ introduce new hyperparameters as hard to calibrate as K itself

Splitting an existing cluster

All options above focus on merging. Splitting is needed when K_init < K_true (under-clustering). Detection heuristics:

  • Bimodal NND distribution within a cluster: if the nearest-neighbour distances of cells in cluster k are bimodal, the cluster contains two spatially segregated populations — a likely split.
  • High intra-cluster violation rate: if many pairs within cluster k have high CF (large territorial overlap), the cluster spans conflicting territories and should be split.
  • Feature sub-structure: fit a 2-component GMM to cells of cluster k alone; if BIC favours 2 components and the two sub-means are spatially non-overlapping, split.
Practical recommendation: implement Option A (overclustering + hierarchical merge) with K_over = 3·K_est where K_est comes from BIC on the feature-only GMM. Run MRF with K_over labels, then merge greedily using feature Mahalanobis distance as primary key and mean territorial CF as a veto (reject merges where mean_CF > τ_low). This adds one loop over K_over² pair candidates — negligible cost for K_over ≤ 30. The resulting K_final is a data-driven estimate of K_true requiring no prior knowledge of the number of cell types.

Combined B→MRF→A pipeline (brainstorm)

Options A and B above are complementary: B fixes under-clustering (spatially fragmented clusters that should be split), A fixes over-clustering (sub-clusters that should be merged). Rather than choosing one, they can be composed into a single pipeline:

Initialize clustering

Any initial partition — Leiden, Ward, or GMM with a rough K_est. The exact method matters less than getting a starting label assignment with reasonable feature coherence. K_init need not equal K_true.

Fit GMM to the initial labels

Fix the cluster assignments from step 1 and fit a K_init-component GMM (e.g. warm-start EM from the empirical means/covariances). This gives posteriors for all cells, enabling the unary term and the frozen-cell mechanism. The GMM does not need to re-cluster — it just needs to assign likelihoods to the existing partition.

Run B: split spatially incoherent clusters

Detect clusters whose cells do not form a single connected component in the spatial graph (Option B, §14). For each fragmented cluster, split along the minimum spatial graph cut into K_frag sub-clusters. Re-fit the GMM components for each new sub-cluster. The label set grows from K_init to K_split ≥ K_init.

Run the MRF algorithm (ICM)

With K_split labels and their GMM posteriors, run ICM with the signed log-ramp pairwise term (§13). This refines cell-level assignments given the now-split initial partition — the spatial boundaries between sub-clusters are sharpened, and cells in violation are corrected.

Run A: merge mosaic-compatible sub-clusters

After ICM, apply the hierarchical merge pass (Option A, §14): greedily merge pairs of sub-clusters that are feature-close (Mahalanobis distance < δ_feat) and do not share territory (mean CF < τ_low). This collapses the K_split over-partitioned label set back toward K_true. The resulting K_final is a data-driven estimate requiring no prior knowledge of K_true.

Run the MRF algorithm again (optional)

The merge pass changes cluster assignments without re-running ICM, so the merged labels may introduce new small violations at merge boundaries. A short second ICM pass (warm-started from the merged assignments) cleans up these boundary artefacts. With a nearly-converged starting state, this typically takes 1–2 sweeps.

Is the second MRF pass actually necessary? The merge step only combines clusters — it never puts two same-type cells adjacent that were not already adjacent. New violations can only arise at the newly-merged cluster boundaries. Whether a second ICM pass meaningfully improves the result depends on how many cells are in those boundary regions and how strong their unary signal is.
Should B be run before or after fitting the GMM? If the initial clustering already has fragmented clusters, the GMM posteriors will be noisy near the fragment boundaries. Running B first (before GMM fitting) gives cleaner sub-clusters for GMM initialization; but then the GMM must be fit to more components (K_split), which is slower and may overfit. Alternatively, run the GMM on K_init and use the posterior gaps to identify which cells are candidates for reassignment before detecting fragmentation.
What if B produces no splits? If the initial clustering happens to be at or below K_true, all clusters are spatially connected and B is a no-op. The pipeline then degenerates to: fit GMM → run MRF → run A. This is essentially the current approach plus a post-merge step. The B pass costs one connected-components check per cluster — negligible.
Does running A after ICM (rather than instead of ICM) produce different merges? ICM at K_split may reassign some cells between sub-clusters, changing the territorial overlap scores used in the merge criterion. In the best case, ICM sharpens the boundaries so that merge candidates have lower mean CF and are more cleanly separable. In the worst case, ICM causes a sub-cluster to absorb cells from a neighboring cluster, making the merge criterion incorrectly veto a beneficial merge.
GMM re-fitting after each split is the expensive step. Each split in step 3 creates new sub-clusters whose covariance structure is a subset of the original cluster. Fitting a full-covariance GMM component to a small sub-cluster (potentially <10 cells in K_over regimes) produces degenerate or near-singular covariances. Regularization (e.g. adding ε·I to Σ) is necessary. Alternatively, the split sub-clusters can inherit the covariance of their parent cluster and only update the mean and weight — a restricted M-step that is much more stable.

Post-merge violator cleanup (brainstorm)

The merge step reassigns labels but does not re-run the MRF. This creates a new class of violations that were invisible at the sub-cluster level but become apparent once K_eff sub-clusters collapse to K final clusters.

Why post-merge violators appear

Two structurally different mechanisms produce them:

Type 1 — wrong-merge artifacts. If the merge criterion accepted a merge between two sub-clusters that belong to different GT types (feature means happened to be close, territorial overlap happened to be below the veto threshold), the resulting merged cluster now contains cells from two spatially separate territories. Cells in one territory are surrounded by cells of the other merged-in territory — high CF, same label. Before the merge, no pair of cells between the two sub-clusters was same-label, so no violation existed. After the merge, every boundary cell between the two sub-cluster territories is in violation.
Type 2 — boundary-exposure artifacts. Even a correct merge (two GT-same sub-clusters joined) can expose new boundary violations. Sub-cluster A tiles the north; sub-cluster B tiles the south; they are correctly merged into cluster 0. A few ICM-misassigned cells from GT-type 1 (the type between them) may have been absorbed into sub-cluster A or B during the ICM phase and were locally harmless — they were not adjacent to any A-or-B cell from the opposite pole. After the merge, cells from A and B are co-labeled, so those ICM errors are suddenly adjacent to cells of the same (wrong) label from the other sub-cluster. Violation score spikes.

The frozen-mask staleness problem

The frozen mask was computed from posteriors of the K_eff-component GMM. After merging, the frozen mask is stale in a specific way:

Before merge: frozen[i] = True iff max_k posterior[i, k] >= conf_threshold This means: cell i is confident it belongs to sub-cluster k. After merge: sub-clusters k and k' are merged into one label. Cell i is still frozen — but its "same-type" neighbors now include cells from sub-cluster k' which may be in a different spatial territory. Result: A frozen cell can now have high-CF same-label neighbors it never had before. The frozen mechanism — originally designed to prevent ICM from touching correct cells — now prevents ICM from correcting cells that are in clear post-merge violations. Concretely: frozen[i] = True, CF(i, j) = 0.7, labels[i] = labels[j] = 0 (merged) Before merge: i and j had different labels — no violation. After merge: same label, CF=0.7 — clear violation. But both cells are frozen → a second ICM pass cannot fix this.

This means a naive second ICM pass (warm-started from merged labels, same frozen mask) will leave all frozen-cell merge violations intact. The frozen mask must be updated to reflect the post-merge situation.

Options for post-merge cleanup

OPTION M1

Second ICM pass with unfrozen boundary cells

After merging, identify every cell involved in a new violation (CF ≥ τ_low with a same-labeled neighbor that was in a different sub-cluster before the merge). Unfreeze these cells. Then run a second ICM pass with the updated frozen mask. Cells not involved in new violations remain frozen — the anchor structure is mostly preserved.

For each merge (A, B) → C: for i in cells(A), j in cells(B): if raw_map.get((min(i,j), max(i,j)), 0) >= tau_low: frozen[i] = False frozen[j] = False Then: run ICM with updated frozen mask (typically 1–3 sweeps from near-convergence)

Pros

  • Directly solves the frozen-mask staleness problem at merge boundaries
  • Unfreezes only cells that have demonstrably new violations — all other frozen cells remain anchors
  • ICM propagates corrections: one unfrozen cell can pull its neighbors via the pairwise term
  • Uses existing ICM machinery — no new algorithm needed

Cons

  • Requires recording which sub-cluster each cell belonged to before the merge — the merge history from merge_history provides this
  • Unfreezing boundary cells removes anchors near the merge boundary: ICM may over-correct if the unary signal in that region is weak
  • Does not help for Type-1 (wrong-merge) violations: if the merge was wrong, ICM will reassign boundary cells but the two erroneously-merged sub-clusters will still share a label
OPTION M2

Post-merge H1 pre-pass + ICM

Apply the H1 force-unfreeze mechanism (§13c) to the post-merge labels. Scan all same-label pairs with CF > θ_hard; for each, unfreeze the less-confident cell. Then run ICM. This is identical to the H1 pipeline in §13c but applied to the merged label assignment rather than the sub-cluster GMM assignment.

Step 1: apply merged labels to all cells Step 2: scan same-label pairs with CF > θ_hard → unfreeze less-confident cell in each pair (H1, §13c) Step 3: run ICM from merged labels with updated frozen mask

Pros

  • Principled: exactly the same logic that was already designed for frozen-cell violations in §13
  • Handles both new post-merge violations and any residual pre-merge violations that ICM left unsolved
  • θ_hard provides a clean threshold distinguishing certain violations (CF > 0.85) from uncertain contacts

Cons

  • θ_hard needs to be conservative: the same-type CF at high density (high_overlap config) reaches 0.50–0.60, so setting θ_hard too low unfreezes legitimate same-tile contacts
  • Less targeted than M1 — does not distinguish between violations caused by the merge and pre-existing ones
OPTION M3

Violator-aware merge: simulate before committing

Before accepting a merge of sub-clusters A and B, simulate the merge and count how many new high-CF same-label pairs it would create. Define a "merge pollution score" as the number of new pairs (i, j) where i ∈ A, j ∈ B, CF(i,j) ≥ τ_low, that would become same-labeled. Accept the merge only if the pollution score is below a threshold.

pollution(A, B) = |{(i,j) : i∈cells(A), j∈cells(B), raw_map.get((min,max),0) >= tau_low}| This is exactly the "all_cfs" count already computed in the merge gate check. Current gate: if len(all_cfs) >= min_adj_pairs AND frac_violating < veto_frac. Stricter variant: also require pollution(A,B) == 0 — no adjacent pairs with CF >= tau_low. This turns the fraction veto into an absolute veto for cross-territory merges.

Pros

  • Preventative: avoids creating violations rather than cleaning them up after the fact
  • The pollution count is already computed as part of the merge veto logic — zero additional cost
  • Naturally handles the wrong-merge case (Type 1): two sub-clusters from different GT types will always have some adjacent pairs with CF ≥ τ_low

Cons

  • A strict zero-pollution requirement can prevent beneficial merges: two correct same-type sub-clusters in a dense mosaic may have a handful of legitimate CF ≥ τ_low contacts at the seam between their territories, which the rule incorrectly vetoes
  • This is essentially a stricter parameterization of the existing fraction veto, not a fundamentally different mechanism
  • Does not address Type-2 (boundary-exposure) artifacts, which arise from ICM errors that are only exposed after the merge — the pollution score was 0 at merge time but becomes non-zero as merged-cluster label is propagated
OPTION M4

Post-merge targeted cell reassignment (without full ICM)

After merging, scan all cells and compute a per-cell violation score: sum of CF(i,j) × [labels[i] == labels[j]] for all spatial neighbors j. Cells with violation score above a threshold are candidates for reassignment. For each candidate, try all other K−1 labels and assign the one that minimizes (unary cost + spatial violation score). Iterate until no candidate exceeds the threshold.

For each cell i: violation_score[i] = Σ_{j∈nbrs(i)} CF(i,j) · [labels[i] == labels[j]] Candidates = {i : violation_score[i] > v_threshold} For each candidate i (sorted by score descending): labels[i] = argmin_k (unary[i,k] + λ · Σ_j w_ij · φ(k, labels[j])) Repeat until |Candidates| = 0 or max_iters reached.

Pros

  • Surgical: only touches cells that are demonstrably in violation — all others are untouched regardless of frozen status
  • Naturally ignores the frozen mask for violations and respects it otherwise — frozen-mask staleness is not a problem
  • Equivalent to a one-at-a-time ICM pass restricted to high-violation cells — low computational cost

Cons

  • Does not propagate corrections: a cell reassigned by this pass may create new violations in its neighbors that are not themselves candidates (their violation score was just below threshold)
  • The threshold v_threshold is a new hyperparameter without a principled default
  • Can oscillate between two states if two neighboring cells each prefer the other's label — needs a maximum iteration count
OPTION M5

Unassign persistent violators (real-data orientation)

Rather than forcing a reassignment, mark cells that remain in high-CF same-label contacts after M1/M2 cleanup as "unassigned" (label = −1 or a reserved sentinel). These cells are flagged as spatially ambiguous — their feature signal says one type, their neighborhood says another. In a real dataset, this is the honest answer: these cells cannot be confidently classified with the available information.

After cleanup pass: persistent_violators = {i : any CF(i,j) >= tau_high and labels[i]==labels[j] for j in nbrs(i)} labels[persistent_violators] = -1 # unassigned For toy data with ground truth: n_unassigned, fraction_correct_if_forced = evaluate_uncertain_cells(...) # are the persistent violators disproportionately true errors?

Pros

  • Honest representation of uncertainty: better to say "unknown" than to guess wrong
  • Useful for downstream analysis: flagged cells can be manually reviewed or excluded from statistics
  • Naturally handles the concentric-ring failure mode (§15) where territorial overlap is structural, not an error

Cons

  • Reduces the useful output: fewer classified cells, lower apparent ARI on toy data
  • Does not improve K recovery — the number of clusters is unchanged, only some cells lose their assignment
  • The τ_high threshold needed here is different from the τ_high used in the ramp (which controls the repulsive saturation point)
Recommended path: M1 (merge-boundary unfreeze) + second ICM pass, falling back to M2 (H1 pre-pass) for residual violations. The two-step strategy:
  1. After merging, record which cells lie on a merge boundary (adjacent pair from different sub-clusters that were merged together).
  2. Unfreeze cells on merge boundaries that have CF ≥ τ_low with their new same-label neighbor — these are the cells the merge created violations for.
  3. Run a second ICM pass (typically 1–3 sweeps from near-convergence).
  4. After ICM, run an H1 scan (CF > θ_hard, same label) on the result. Any remaining violations at this threshold are near-certain errors that escaped M1 — unfreeze the less-confident cell in each and run one more ICM sweep.
This sequence is cheap (second ICM starts from near-convergence), targeted (only merge-boundary cells are unfrozen in the first pass), and principled (H1 in the second pass uses the same logic as §13c). The M3 preventative approach is also worth folding in: tighten the merge veto to require zero adjacent pairs with CF ≥ τ_low (rather than a fraction), which eliminates Type-1 wrong-merge artifacts before they occur and reduces the cleanup burden downstream. M5 (unassignment) is not recommended as a primary step for toy data but is the right framing for production use where honest uncertainty quantification matters more than maximizing classified-cell count.
How many post-merge violations are actually of Type 1 vs Type 2? In the current toy configs (anisotropic, moons), the merge errors are diagnostic. Looking at the merge dendrogram: if a merge connects two sub-clusters of different GT majority types, it is by definition a Type-1 event. If both sub-clusters share the same GT majority but the post-merge violation count is higher than expected, it is Type 2. Instrumenting the merge step with per-merge violation-delta tracking would answer this directly.
Does the second ICM pass help or hurt for correct merges? For a correct merge of two same-GT sub-clusters, the second ICM pass should be a no-op (or improve a few ICM-misassigned boundary cells). For a wrong merge, the second ICM pass will partially undo the wrong merge by reassigning boundary cells — but the cluster-level label is unchanged. The cells may move but the wrong merge persists in the label space. This asymmetry suggests M3 (preventing wrong merges) has higher leverage than cleanup.

15. Non-Gaussian clusters: moons and circles

GMM unary terms assume each cell type occupies a convex, ellipsoidal region in feature space. Moon-shaped or ring-shaped distributions (e.g. if cells are characterised by an angle or a ratio) produce poor GMM fits: either one type is split across multiple components, or two distinct types are merged because their convex hulls overlap.

Why moons are hard for the current pipeline

The unary term actively works against non-Gaussian shapes. A cell on the outer arm of a moon-shaped cluster has lower posterior for its own type than for the nearest ellipsoidal type. Freezing is based on this posterior, so outer-arm cells are frozen with the wrong label — and ICM can never fix them because frozen cells are never updated. Worse, the spatial term sees these cells as territorial violations and pushes them further toward the wrong label.

Can overclustering + merging (§14) address this?

Yes — this is the most promising path, and it works in a post-step without touching the MRF inference core.

Moon decomposition via overclustering: True cluster shape: moon (two Gaussian arms connected) ↓ K_over GMM: arm-1 → sub-clusters {A1, A2} arm-2 → sub-clusters {B1, B2} (each sub-cluster is locally Gaussian) ↓ After MRF: spatial territories of A1, A2, B1, B2 are refined ↓ Merge step: feature distance A1–A2 < δ_feat, no territorial overlap → merge feature distance B1–B2 < δ_feat, no territorial overlap → merge feature distance (A1∪A2)–(B1∪B2) < δ_feat, no territorial overlap → merge ↓ Result: one merged cluster covering the full moon shape

The key insight is that the mosaic constraint provides the non-Gaussian structure for free: if the moon-shaped type genuinely occupies a single spatial territory, the territorial non-overlap criterion will correctly allow the arms to be merged. The feature-only GMM cannot distinguish the arms from other types; the spatial structure can.

Where it still fails

Concentric rings: the territorial-overlap criterion breaks. If two ring-shaped types are concentric (one ring inside the other), every cell of the inner ring is geographically surrounded by cells of the outer ring. Territorial overlap between the two types is extreme (CF ≈ 1 everywhere) — but this is correct, not a mosaic violation. The MRF will try to eliminate the "violation" and the merge step will refuse to merge (high territorial overlap). Both steps actively work against the correct solution. This is a fundamental limitation: the mosaic prior assumes territorial exclusivity, which concentric rings violate by design.
Over-merging: feature similarity alone can bridge wrong sub-clusters. In a high-overlap config, sub-clusters from two different types may be feature-similar (their cluster means are close) and also happen not to overlap territorially in the overcluster assignment. The merge criterion would incorrectly merge them. A stronger veto — require that the merged cluster passes a spatial regularity check (NND-CV and inter-cell distance distribution consistent with a single Poisson-disk mosaic) before accepting the merge — reduces this risk.
ALTERNATIVE

Kernel PCA or UMAP embedding before GMM

Apply a non-linear dimensionality reduction (kernel PCA with RBF kernel, or UMAP) to the feature matrix X before fitting the GMM. Non-Gaussian clusters in the original space become approximately Gaussian in the embedding, making the GMM unary term valid. The MRF spatial term is unchanged (it operates on polygons, not features). The main risk is that the embedding destroys the metric structure the Mahalanobis unary term relies on — the posterior gaps become less interpretable as nats.

Pros

  • Directly solves the Gaussian assumption without changing the MRF framework
  • UMAP preserves local structure — nearby cells in feature space remain nearby in the embedding

Cons

  • UMAP/kPCA are stochastic and sensitive to hyperparameters (n_neighbors, perplexity)
  • Embedding quality degrades with fewer cells; toy datasets have N≈200 which is near the boundary for UMAP reliability
  • Posterior probabilities in embedding space are not calibrated — the freezing and unary-cost logic needs revisiting
ALTERNATIVE

Normalizing flows as the GMM replacement

Replace the GMM with a normalizing flow model — a learnable bijection f: X → Z such that Z is Gaussian-distributed per component. Normalizing flows can represent arbitrary densities, so moon and ring shapes are handled exactly. The unary cost ψ_i(k) = −log p_k(X_i) is still well-defined and finite everywhere. The MRF and ICM layers are unchanged. The cost is implementation complexity and training time (flows require gradient-based optimisation, not a closed-form EM step).

Summary for non-Gaussian clusters: the overclustering + hierarchical merge approach (§14, Option A) handles moons and multi-arm shapes well, provided the arms are spatially separated (occupy distinct territories). Concentric or radially symmetric configurations require a different prior than mosaic exclusivity and are outside the scope of the current formulation. For real RGC data, moons and rings are unlikely; the bigger concern is elongated anisotropic clusters, which are already handled by the GMM full-covariance option.

16. Post-ICM residuals: why steps 6–7 fail and what to do instead

After ICM converges (step 5b), a small number of cells remain incorrectly assigned. Steps 6 (pairwise conflict resolution) and 7 (residual violator assignment) were designed to clean these up, but in practice they rarely help and sometimes make things worse. This section diagnoses why and proposes better alternatives.

The failure mode

Feature impostors with spatial confirmation. The hardest residual cases are cells that are (a) feature-closer to the wrong cluster, and (b) have genuine polygon overlap with cells from the wrong cluster. ICM assigns them to the wrong cluster because the unary term dominates, and the pairwise term confirms rather than corrects: the overlap with the wrong-cluster neighbours looks like evidence that the cell belongs there. From the global spatial picture, however, it is obvious that these cells break the mosaic — they sit inside the wrong territory.
Why steps 6–7 can't fix this. Pairwise conflict resolution (step 6) looks for pairs of same-label cells with repulsive contacts and jointly finds a lower-energy assignment. But if the feature impostor is frozen, or if its neighbourhood energy consistently prefers the wrong label, the optimizer finds the current assignment already optimal — there is no lower-energy joint solution. Residual violator assignment (step 7) then assigns −1 or leaves the cell unchanged. The issue is not a missing algorithm — it is missing information: the pairwise term as currently designed cannot signal that a cell is in the wrong territory, only that it has unexpected overlap with a same-label neighbour.

Proposed approaches

PATHWAY 1

Global spatial regularity as an additional unary term (W11 improvement)

The current W11 cell-count regulariser (β · |n_k − n_expected|²) only encodes one global property: cluster sizes should be equal. A stronger version would encode spatial regularity: cells assigned to cluster k should collectively form a pattern consistent with a Poisson-disk mosaic — regular spacing, roughly uniform density, no large voids or dense islands.

One practical implementation: after each ICM sweep, compute the NND-CV (nearest-neighbour distance coefficient of variation) and the fraction of cells in the largest connected component for each cluster. Append a cluster-level penalty to the unary term — cells on the wrong side of a spatial regularity boundary pay extra. This does not require solving anything globally; it is a per-sweep update of a proxy regulariser, similar to how count regularisation works.

A lighter version: penalise cells whose assigned cluster has a disconnected spatial footprint. If cluster k has two separate connected components (identified via k-NN graph within the cluster), each cell in the smaller component pays a penalty proportional to the component size ratio. This encodes the mosaic requirement that each type occupies a single contiguous territory.

Pros

  • Principled: directly encodes the mosaic prior rather than patching it
  • Compatible with ICM — can be added as a per-cell unary term update without changing the optimisation structure
  • Component penalty is cheap to compute (BFS on the k-NN graph)

Cons

  • Penalty strength (β) is a new hyperparameter requiring calibration
  • Dynamic unary updates (recomputed each sweep) break the monotone energy descent guarantee of standard ICM unless β is kept small
  • Connected-component topology changes discretely — creates energy landscape cliffs that can cause oscillation near transitions
PATHWAY 2

Polygon dilation before coverage-fraction computation

The current coverage fraction (CF) is zero for cells whose polygons are exactly touching but not overlapping — a common configuration for well-separated same-type neighbours in a tight mosaic. These pairs contribute no pairwise signal at all, even though geometrically they are the clearest same-tile contacts. Slightly inflating each polygon by a fraction of its mean diameter (e.g. 5–10%) before computing CFs would:

  1. Convert near-zero "touching" contacts into small positive CF values, giving the signed ramp an attractive contribution for genuine same-tile pairs
  2. Amplify the distinction between intra-tile contacts (CF rises from ~0 to ~0.05) and inter-tile violations (CF rises from ~0.3 to ~0.35) without changing their relative ordering

Alternatively, add a proximity term that contributes an attractive weight for pairs within a smaller radius even when CF=0, using the existing _build_proximity_map already implemented in leiden_mosaic.py. A convex combination of CF-based weight and proximity weight: w_ij = α·CF_weight(i,j) + (1−α)·proximity_weight(i,j) would catch the "touching but not overlapping" case.

Pros

  • Addresses a real geometric gap: touching-but-not-overlapping pairs are currently invisible to the spatial term
  • Proximity map already exists in the codebase — the combination approach is a low-code change
  • Makes the attractive zone of the signed ramp more effective for correctly-assigned cells

Cons

  • Dilation radius is a new hyperparameter
  • Dilation can create artefactual overlaps between non-adjacent cells in dense mosaics, raising CF for genuinely non-touching pairs
  • The proximity term conflates distance and overlap — two elongated cells can be close but non-overlapping, or distant but highly overlapping; mixing the signals may add noise
PATHWAY 3

Per-cluster adaptive tau: penalise unexpectedly large CFs

The current τ_low is a single global threshold calibrated from the overall same-label CF distribution. But each cluster has its own intra-tile CF baseline: a cluster of large cells at high density will have higher typical CFs than a sparse cluster of small cells. A cell with CF=0.25 to a same-label neighbour is normal for one cluster and a strong violation for another.

Proposal: calibrate τ_low per cluster as the p99 of same-label CFs within that cluster's frozen cells only. The ramp weight for pair (i, j) is then:

w_ij = ramp(CF_ij, τ_low[y_i], τ_high[y_i]) if y_i == y_j (or the average of the two cluster thresholds)

A variant that requires no threshold at all: use the z-score of CF_ij within the current cluster's CF distribution as the pairwise weight. A CF that is 3σ above the cluster mean is a much stronger signal than one that is only 0.5σ above. The ramp is replaced by a continuous, data-adaptive weight.

Pros

  • Directly addresses the feature-impostor problem: a cell whose CF with a same-label neighbour is anomalously high (relative to that cluster) is almost certainly misassigned
  • Makes the spatial penalty scale-invariant across clusters
  • Compatible with the existing calibration infrastructure — τ_low is already computed per quantile

Cons

  • Requires enough same-label frozen pairs per cluster for a reliable distribution estimate (minimum ~10 pairs)
  • Per-cluster thresholds reintroduce label-dependency into the pairwise term, which can create discontinuities when labels flip during ICM
  • Early ICM iterations use noisy label estimates, so per-cluster CF distributions are initially unreliable — may need to be recomputed only after ICM stabilises (connection to W2b iterative recalibration)
PATHWAY 4

Re-run the full pipeline after merging (EM-like iteration)

After the split-merge cycle completes, the label assignment is better than the initial GMM but not perfect. A natural extension: use the post-merge labels as warm-start GMM means for a second GMM fit, then run ICM again. This mirrors the EM algorithm — alternating between a model update (re-fit GMM to the current assignment) and an inference step (ICM).

Concretely:

  1. Take the post-merge labels as the initial assignment
  2. Re-fit a K-component GMM with init_params="random" but means initialised at the post-merge cluster centroids
  3. Recompute unary costs from the new GMM
  4. Re-run ICM with the new unary (skip the split and merge steps this time)

One iteration of this loop is already partially implemented as recalibrate=True (which re-estimates tau/lam from post-ICM labels). A full re-GMM step is stronger: it updates the Gaussian means and covariances to reflect the spatially-corrected assignment, giving cells that were on the boundary more accurate unary costs for the second ICM.

Pros

  • EM convergence theory guarantees the joint GMM+MRF energy does not increase across iterations (given a well-defined joint objective)
  • The re-fitted GMM can correct feature impostors: a cell that was assigned to the wrong cluster in round 1 shifts the wrong cluster's mean slightly; in round 2, it is less likely to fit that cluster, making the unary correction more effective
  • Does not require any algorithmic changes to the existing pipeline — just a loop in fit()

Cons

  • Wall-clock time scales linearly with the number of iterations — expensive for large datasets
  • Risk of degenerate cycles: a cell oscillates between two labels across iterations if both assignments are locally consistent
  • Re-fitting the GMM on a partially-corrected assignment can collapse components if too many cells from one cluster were reassigned in round 1
  • The joint objective (GMM likelihood + spatial energy) is not guaranteed to converge if the GMM E-step uses hard assignments (MAP labels) rather than soft ones
PATHWAY 5 (new)

Replace steps 6–7 with territory-aware label propagation

Instead of trying to fix residuals by solving small sub-problems (steps 6–7), propagate labels from high-confidence spatial anchors outward using a graph diffusion / label spreading approach. Treat frozen cells as labelled seeds and all unfrozen cells as unlabelled. Run semi-supervised label spreading on the spatial CF graph: each unfrozen cell is assigned the label that minimises the total edge cost to its labelled neighbourhood.

This is stronger than ICM because it does not optimise one cell at a time in isolation — it considers the entire connected territory of each cluster simultaneously. Cells that are "inside" the wrong territory get pulled toward the correct label by the entire surrounding spatial context, not just their immediate neighbours.

Implementation: use sklearn.semi_supervised.LabelSpreading with the CF-based affinity matrix, with frozen cells as the labelled set. Only apply to unfrozen cells that are still in violation after ICM, to avoid destabilising well-converged cells.

Pros

  • Captures the global territorial context that ICM misses — a cell surrounded by the wrong cluster's territory is corrected even if its immediate neighbours are ambiguous
  • Label spreading has well-understood convergence and is differentiable, enabling future gradient-based tuning of the affinity kernel

Cons

  • Semi-supervised spreading ignores the feature unary entirely — purely spatial, which can over-correct cells with legitimately ambiguous feature assignments
  • The affinity matrix (from CF values) must be positive semi-definite for label spreading; CF-based matrices are not guaranteed to be PSD
  • Interaction with the frozen mask needs careful design — spreading from wrong-label frozen cells will propagate errors

Recommendation

Priority order:
  1. Pathway 3 (per-cluster adaptive tau) — highest expected return for lowest implementation cost. The signal is directly informative about the feature-impostor problem. Can be combined with the existing iterative recalibration (W2b) in a single pass.
  2. Pathway 4 (re-run after merge) — one additional GMM + ICM cycle at almost no code cost. The EM interpretation is principled. Implement as n_em_iters=1 parameter; default 0 for backward compat.
  3. Pathway 2 (polygon dilation / proximity term) — useful if the main failure mode is "touching but zero CF" pairs. Easy to validate: compare the CF distribution before and after dilation on a known-good mosaic.
  4. Pathway 1 (spatial regularity penalty) — most principled long-term direction. Start with the connected-component penalty (cheapest version) as a unary add-on before the second ICM pass after merge.
  5. Pathway 5 (label spreading) — most powerful but most complex; requires validating that the CF affinity matrix does not propagate errors from frozen impostors.

17. Stochastic cleanup: escaping oscillation with tracked best-state selection

The observed problem

The alternating cleanup pipeline (P3 conflict resolution → P4 residual violators → EM re-fit → repeat) empirically decreases the total MRF energy at each round. The post-cleanup EM step confirms this: running EM after cleanup does not hurt and occasionally helps, which means the cleanup genuinely found a lower-energy state that EM then consolidated. However, the sequence can also visit a bad transient state — cleanup moves a cell spatially, EM re-fits the model and gives the moved cell high unary cost in its new cluster, and the next ICM pass flips it back. This cycle can continue indefinitely without converging.

The feature–spatial tension loop. Cleanup overrides the feature model (to fix mosaic violations). EM then re-anchors features to the post-cleanup state. If cleanup placed a cell in a cluster whose feature distribution the cell barely fits, EM learns a Gaussian that excludes it, ICM reverts it, and the next cleanup re-moves it. The two optimisers are minimising different objectives in turns rather than jointly converging.

Empirical signal: the total loss does decrease

The observation that n_em_iters_post_cleanup > 0 never hurts and sometimes helps suggests that the joint MRF energy (unary + λ × pairwise violations) is not oscillating — cleanup brings it down, EM consolidates it further. The instability is localised to individual cells near the feature–spatial boundary, not to the global energy. This means:

  1. The cleanup+EM sequence visits states of monotonically decreasing average energy.
  2. Individual cleanup passes may temporarily increase energy for a few cells while reducing it globally.
  3. A "best state seen so far" tracker would never regress globally even when local oscillations occur.

Proposed approaches

APPROACH A

Tracked iterated cleanup (best-energy selection)

Run n cleanup+EM cycles. After each cycle, compute the total MRF energy E(labels) = Σ unary[i,k_i] + λ Σ w_ij·[k_i == k_j]. Keep a copy of the labels at the cycle with the lowest energy. Return that copy, not the final iterate.

labels_best ← labels_init E_best ← E(labels_init) for t in 1..n_cleanup_iters: labels ← cleanup_step(labels) # one conflict-res / residual-viol pass labels ← em_step(labels) # one EM iteration E ← E(labels, nbrs, lam) if E < E_best: E_best, labels_best ← E, copy(labels) return labels_best

This is safe by construction: it can only return a state with energy ≤ the starting energy, regardless of how badly the iterates oscillate. The extra cost is one _compute_energy call (already O(n)) per cycle.

Pros

  • Zero additional hyperparameters — uses the MRF energy that is already tracked
  • Monotone guarantee on the returned energy
  • Handles oscillations automatically without detecting them
  • Trivially extensible: increase n_cleanup_iters for more exploration

Cons

  • Memory overhead of one labels copy per cycle (negligible)
  • Does not escape local minima — still a local search, just more robust
  • The MRF energy may not perfectly reflect the true mosaic quality (e.g. it cannot penalise large connected components spanning two territories)
APPROACH B

Stochastic ICM with temperature annealing inside cleanup

Replace the deterministic argmin in the cleanup ICM passes with a Boltzmann sample: accept label k for cell i with probability proportional to exp(−E_local(i,k) / T). Start with high T (allows uphill moves to escape the feature–spatial trap) and anneal toward T → 0 (greedy convergence). Track the best state as in Approach A.

P(k_i = k) ∝ exp(−[unary[i,k] + λ·Σ_j w_ij·[k_j == k]] / T(t))

This is simulated annealing applied specifically to the cleanup passes, not to the full ICM. The main ICM remains deterministic (Gauss-Seidel). Only the post-cleanup correction loop samples stochastically.

Pros

  • Can in principle escape the feature–spatial trap by temporarily accepting a worse feature fit
  • With a good annealing schedule, converges to a solution near the global optimum of the joint energy

Cons

  • Temperature schedule is a new hyperparameter requiring tuning per dataset
  • Non-deterministic — results vary across runs even with fixed random seed
  • Significantly slower than deterministic cleanup for the same number of cycles
APPROACH C

Perturb-and-fix with iterated local search

After each cleanup+EM cycle converges locally, deliberately perturb the solution: randomly flip a small subset of boundary cells (those near mosaic violations) to their second-best label, then re-run cleanup+EM from this perturbed state. Accept the new state if its energy is lower; otherwise revert. This is iterated local search applied to the MRF problem.

for t in 1..n_restarts: labels_perturbed ← perturb(labels_best, frac=0.05) # flip 5 % of violators labels_candidate ← cleanup_em(labels_perturbed) if E(labels_candidate) < E(labels_best): labels_best ← labels_candidate

Pros

  • Can escape deep local minima that Approach A cannot, by using random restarts from a nearby state
  • The perturbation step is cheap (flip a few cells, not a full GMM restart)

Cons

  • Adds a perturbation-fraction hyperparameter
  • Risk of perturbing well-placed cells and never recovering them
  • More complex than Approach A; harder to tune

Recommendation

Start with Approach A. It requires zero new hyperparameters, has a monotone guarantee, and costs almost nothing to implement over the existing tracked-energy infrastructure (_compute_energy is already called in _fit_single). Add a n_tracked_cleanup_iters parameter (default 0, disabled) that replaces the fixed n_cleanup_steps + n_em_iters_post_cleanup loop with a best-tracking version. Only move to Approach B or C if A does not resolve the oscillation — they have more implementation cost and tuning burden.

18. Making _step_cleanup stochastic: concrete mechanisms

The empirical motivation

With the tracked best-energy loop implemented, the observed energy trajectory is always:

E[0] = −157.5 *E[1] = −526.1 E[2] = −526.1 …

The cleanup+EM cycle reaches a fixed point after exactly one iteration. The tracker detects this and stops early, confirming there is no oscillation to escape — the pipeline is fully deterministic. This is actually good news: cleanup genuinely improves the energy, and converges immediately. The downside is that there is only one outcome regardless of how many iterations you run. To explore multiple local optima, the cleanup itself must become stochastic.

What "stochastic cleanup" would buy. If each tracked iteration produces a different local optimum (because small random decisions during conflict resolution lead to different swaps), the tracker can compare multiple optima and keep the best. The cost per iteration is the same; the benefit is a wider search over the local neighbourhood of the MRF energy landscape.

Concrete mechanisms, from simplest to most principled

MECHANISM 1

Weight jitter — randomise conflict processing order

In _resolve_conflicting_pairs, pairs are currently sorted by descending weight and processed in that fixed order. Processing pair (i, j) first changes the neighbourhood energy for pairs that share a cell with i or j, affecting whether those pairs accept a swap. Adding small Gaussian noise to the sort key breaks the determinism:

w_eff(i, j) = w(i, j) + ε · N(0, 1) pairs sorted by w_eff (descending) ← different each call

With ε ≈ 0.05 × median_weight, the order is nearly the same as before but occasionally pairs near the boundary swap position, leading to different conflict resolutions. Temperature is controlled by ε; ε = 0 recovers the original deterministic behaviour.

Pros

  • One-line change inside _resolve_conflicting_pairs
  • Continuous temperature parameter (ε)
  • Fully controlled by random_state

Cons

  • Weak perturbation — many pairs are far from each other so order changes rarely matter
  • Does not break the fixed point if all conflicts are isolated (no shared cells)
MECHANISM 2

Boltzmann acceptance — allow uphill swaps with decreasing probability

Currently a pair swap is accepted if and only if the joint energy decreases. Replace this with a Metropolis-style acceptance criterion:

ΔE = E_proposed − E_current P(accept) = 1 if ΔE ≤ 0 exp(−ΔE / T) if ΔE > 0

With small T, this is nearly greedy. With larger T, some uphill moves are accepted, allowing the cleanup to escape shallow local optima. T can be annealed across tracked iterations: high T on the first pass, low T on the last. The tracker keeps the best energy seen across all passes.

Pros

  • Principled: Metropolis-Hastings applied to the 2-cell sub-problem
  • Temperature T is interpretable in the same units as the MRF energy (nats)
  • Annealing schedule can be tuned independently of the tracker

Cons

  • Accepts swaps that increase energy → individual passes may regress; need the tracker to recover best state
  • T requires calibration: too high = random walk; too low = same as deterministic
  • Natural scale: T ≈ 10–50 % of median unary gap (so that a "borderline" impostor occasionally gets corrected)
MECHANISM 3

Unary noise injection — flatten the energy landscape temporarily

Before each cleanup pass, add a small Gaussian perturbation to the unary costs used during conflict resolution and residual-violator assignment:

unary_noisy[i, k] = unary[i, k] + σ · N(0, 1)

The noisy unary is used only inside the cleanup pass; the original unary is used to compute the final energy for tracking. σ acts as a temperature: with σ = 0, deterministic; with σ ≈ 0.5 × median_unary_gap, the label preference for boundary cells is occasionally reversed, allowing different cells to be corrected. After the pass, the tracker compares the resulting assignments by true energy (without noise) and keeps the best.

Pros

  • Directly targets the root cause: the unary is what locks cells into wrong local optima
  • Easy to implement: one call to np.random.normal before the cleanup step
  • The true energy (without noise) is the correct comparison metric, and the tracker already computes it
  • Can be applied to both conflict resolution and residual violators without changing their logic

Cons

  • Noisy unary may cause correct cells to be swapped as well as incorrect ones — the tracker must catch any regressions
  • σ requires calibration: a natural default is σ = lam_alpha × median_unary_gap × 0.2 (20 % of the calibrated flip threshold)
MECHANISM 4

Random cell perturbation before cleanup (ILS)

Before each tracked cleanup iteration, randomly flip a small fraction of boundary cells (cells with any repulsive same-label contact, i.e. current violators plus their neighbours) to their second-best label by unary cost. Then run the full cleanup from the perturbed state.

cells_to_perturb = random.sample(violators, k = max(1, frac × |violators|)) for c in cells_to_perturb: labels[c] ← argmin_{k ≠ current_k} unary[c, k] # second-best label

This is Iterated Local Search (ILS) applied to the MRF problem. The perturbation gives the cleanup a different starting point each iteration; the tracker keeps the best endpoint. frac controls how large the perturbation is — start with 0.1–0.2 (10–20 % of violating cells).

Pros

  • Largest perturbation of the four — most likely to escape deep local optima
  • The cleanup acts as the "local search" phase, always trying to improve from the perturbed state
  • No new parameters beyond frac (perturbation fraction)

Cons

  • May perturb correctly-placed cells, requiring more cleanup iterations to recover
  • The perturbed starting point may be further from any good local optimum, making cleanup less effective
  • Relies on the tracker to filter out bad outcomes — must run more tracked iterations than the deterministic version
MECHANISM 5

Stochastic unfreezing — probabilistic relaxation of the frozen mask

This mechanism addresses a different root cause from Mechanisms 1–4: incorrectly frozen cells. A cell is "incorrectly frozen" when it has high GMM posterior for the wrong cluster (feature impostor) and therefore sits in the wrong territory as a hard anchor. Because frozen cells are excluded from ICM, conflict resolution, and residual violators, the local optimum of the cleanup is partly defined by where the frozen cells sit — no amount of stochastic perturbation to the unary or conflict ordering can move a frozen cell.

The theta_hard H1 pre-pass already handles extreme impostors (CF > θ_hard with a same-label neighbour), but an impostor with 93 % GMM confidence and only moderate spatial violation passes through still frozen.

Proposed mechanism: during each tracked cleanup iteration, randomly unfreeze a small fraction of borderline frozen cells — those with posterior in the range [conf_threshold, conf_threshold + δ] — and include them as candidates for conflict resolution:

borderline = {i : conf_threshold ≤ max_posterior[i] < conf_threshold + δ} to_unfreeze = random.sample(borderline, k = max(1, p_unfreeze × |borderline|)) frozen_stoch[to_unfreeze] ← False # only for this cleanup iteration

These cells are unfrozen for one cleanup pass only; the original frozen mask is restored before the next tracked iteration. The tracker evaluates the energy with the stochastically unfrozen state and keeps the best. The parameter δ controls how far down the confidence ladder to reach (e.g., δ = 0.05 means cells with 90–95 % confidence are eligible). p_unfreeze controls how many are actually unfrozen per pass (e.g., 0.1–0.3).

Pros

  • The only mechanism that can correct feature impostors without requiring theta_hard to be tuned
  • Complementary to Mechanisms 1–4 — can be combined with unary noise for doubly stochastic passes
  • The original frozen mask is not permanently changed; the tracker protects against bad unfreezing outcomes

Cons

  • Unfreezing a correctly-placed cell can degrade the result — the tracker must run enough iterations to recover
  • Two new hyperparameters: δ (how borderline) and p_unfreeze (what fraction)
  • Interacts with the frozen-for-model snapshot: if the best tracked state has a different frozen mask, the model dict needs to reflect the right one for downstream diagnostics

Recommendation and interaction with the tracker

Two-layer stochastic cleanup: Mechanisms 3 + 5.

Layer 1 — Mechanism 3 (unary noise): targets unfrozen feature impostors — cells that could be moved but whose unary cost locks them in. Add cleanup_sigma: float = 0.0 to _step_cleanup; if σ > 0, sample noise = rng.normal(0, σ, unary_final.shape) and pass unary_final + noise to conflict resolution and residual violators only. Energy is always evaluated on the original unary — the tracker is fair. Default σ ≈ 0.2 × lam_alpha × median_unary_gap.

Layer 2 — Mechanism 5 (stochastic unfreezing): targets frozen feature impostors — cells the current logic cannot touch at all. Each tracked iteration also randomly unfreezes a fraction of borderline-frozen cells (posterior in [conf_threshold, conf_threshold + 0.05]). The original frozen mask is restored before the next iteration; the tracker absorbs any bad outcomes.

Mechanism 1 (weight jitter, ε ≈ 0.05 × median_weight) adds diversification of conflict-processing order at negligible cost and can be layered on top of both. Mechanism 2 (Boltzmann acceptance) and Mechanism 4 (cell perturbation) are stronger but require more iterations to benefit from them; try Mechanism 3 first.
